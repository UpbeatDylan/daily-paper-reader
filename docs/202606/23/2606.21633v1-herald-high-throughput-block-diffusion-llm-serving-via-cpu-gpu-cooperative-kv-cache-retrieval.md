## 摘要
扩散LLM（dLLM）通过每次前向传播生成多个token，相比自回归解码提高了GPU利用率，但其KV缓存仍随上下文线性增长，限制了长上下文下的吞吐量。将KV缓存卸载到主机DRAM可缓解这一内存压力，但受限于有限的PCIe带宽，仅需召回稀疏的KV子集。在块dLLM中，相关的KV条目在块内的去噪步骤间保持一致，通过一次性识别top-k条目并在所有去噪步骤中重复使用，可实现高精度选择。这一特性对卸载具有吸引力，因为它将选择开销分摊到整个块上，但需要对完整KV缓存进行精确注意力计算，这在卸载场景下成本过高。我们提出了HERALD，一种面向块dLLM的KV卸载系统，通过两个机会解决了这一问题：将所需的选择计算量减少一个块大小的因子，并使选择与去噪重叠。在三种块dLLM和五个长上下文任务上，HERALD在5-10%的KV预算下实现了近乎无损的准确率，每块延迟降低最高1.59倍，吞吐量相比纯GPU推理提升最高2.47倍，且加速效果随上下文长度增长而增加。

## Abstract
Diffusion LLMs (dLLMs) improve GPU utilization over autoregressive decoding by generating multiple tokens per forward pass, but their KV cache still grows linearly with context, limiting throughput at long contexts. KV cache offloading to host DRAM alleviates this memory pressure, but the limited PCIe bandwidth necessitates recalling only a sparse subset of KV entries. In block dLLMs, the relevant KV entries remain consistent across denoising steps within a block, enabling high-accuracy selection by identifying the top-k entries once and reusing them throughout all denoising steps. This property appears attractive for offloading as it amortizes the selection overhead across the entire block, but it requires exact attention over the full KV cache, which is too expensive under offloading. We present HERALD, a KV offloading system for block dLLMs that resolves this through two opportunities that reduce the required selection compute by a factor of the block size and enable selection to be overlapped with denoising. Across three block dLLMs and five long context tasks, HERALD achieves near-lossless accuracy at 5-10% KV budget and up to 1.59x lower per block latency and 2.47x higher throughput over GPU-only inference, with speedups growing with context length.

# HERALD 论文详细中文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **问题背景**：扩散 LLM（dLLM），特别是块扩散 LLM（block dLLM），通过每步生成多个 token 来提高 GPU 利用率，克服自回归解码的算术强度低、GPU 利用率不足的问题。然而，其 KV 缓存仍随上下文长度线性增长，在长上下文场景下（如 32K token）迅速占满 GPU HBM 内存，导致批处理大小受限、吞吐量严重下降。
- **现有方案不足**：将 KV 缓存卸载到主机 DRAM 可缓解内存压力，但 PCIe 带宽（约 64 GB/s）远低于 GPU 内存带宽（3.35 TB/s），全量传输延迟远超模型前向时间。现有卸载系统（如 InfiniGen、MAGE 等）通过召回稀疏 KV 子集来减少传输量，但面临选择精度和延迟的权衡。
- **块 dLLM 的特殊机遇**：块内去噪步骤间注意力模式具有强时间一致性——第一步的 top-k 条目与后续步骤高度相似。利用此特性，可一次性选择 top-k 并跨步重用（准确率 82-88%），将选择开销分摊到整个块。但实现该精确选择需要完整 KV 缓存的注意力计算，在卸载场景下成本过高。
- **核心挑战**：
    1.  **硬件映射挑战**：全 KV 注意力计算可在 CPU（靠近数据但计算弱）或 GPU（计算强但需 PCIe 传输）执行，两者都面临严重瓶颈。
    2.  **依赖挑战**：选择操作位于每个块的 critial path，且下一块的选择依赖当前块完成后的最终 token 前缀，无法与当前块去噪重叠。

## 2. 方法论：核心思想、关键技术细节

### 2.1 两个关键机会

- **机会1：单个 [MASK] 查询足以进行块级选择**
    - 在块0时刻，所有 token 均为 [MASK]，共享相同的前 RoPE 嵌入，仅因 RoPE 位置编码不同。中心位置（p_start + floor(B/2)）的查询能最小化与所有前缀 token 相对偏移的均方偏差，从而使单一查询代表整个块。
    - 实验证明：B=32 时，单个中心 [MASK] 查询可恢复完整 B 查询选出 top-k 的 84%；B=64 时仍达 79%。同时，将 CPU 注意力计算量减少 B 倍，获得 22-24x 加速，足以在 CPU 上执行全 KV 注意力。

- **机会2：Step-0 Logits 提供免费语义代理**
    - 在块的第0步，模型已对所有 B 位置产生 logits（基于完整前缀上下文）。尽管调度器仅提交置信度高的少量位置，但所有位置的 argmax 解码构成了整个块的“草稿”（draft），其 KV 缓存可作为下一块选择的前缀代理。
    - 该草稿虽与最终块不完全一致，但保留了足够的语义信息，使下一块的 top-k 选择几乎不受影响（仅需增加少量 KV 预算即可恢复准确率）。这打破了数据依赖，允许选择与当前块后续去噪重叠。

### 2.2 HERALD 系统设计

- **双流解码管道 + 双缓冲稀疏 KV 池**：
    - **主流（Main Stream）**：GPU 负责当前块的去噪。
    - **检索流（Retrieval Stream）**：CPU 与 GPU 协作，为下一块执行 KV 选择。主流在每一步产生草稿后触发检索流，CPU 利用单中心 [MASK] 查询扫描完整 KV 缓存，执行融合的在线 softmax 和 top-k 选择，将选择的条目传输到 GPU 空闲池。
- **CPU-GPU 协同 KV 检索**：
    - 每层处理三阶段：GPU QKV 投影 → CPU 融合注意力+top-k 选择（利用在线 softmax 统计，聚合 G 个查询头为单一重要性得分，减少 top-k 遍数） → GPU FFN。
    - 随机读取分散的 KV 条目到连续缓冲区的 gather 操作与 GPU 执行重叠，DMA 传输与下一层 CPU 选择重叠，整体检索流延迟 = CPU 选择 + GPU FFN 时间。
- **优化 CPU 内核**：
    - 融合注意力与 top-k：流式逐 tile 处理 KV 缓存，同时维护每个查询头的在线 softmax 统计；在单个 KV 头处理完后，利用已计算的统计量聚合所有查询头得分为单一重要性得分，再用最小堆提取 top-k，将 top-k 遍数从 G 降至 1。
- **Prefill 阶段**：处理提示词后，KV 缓存逐层异步复制到 CPU DRAM；然后执行一次类似块0第一步的轻量前向，并对提示词 KV 进行 top-k 选择，预热稀疏池。

### 2.3 执行流程（以块 N 为例）

1.  Step 0：GPU 前向，执行置信度感知的掩码解开。
2.  生成草稿：对所有 B 位置 argmax，填充草稿块。
3.  合并步1与草稿前向：融合为 2B token 的前向，自定义注意力掩码（两组互不关注）。
4.  **并行**：
    - **主流**：继续去噪步骤 (2..T)，逐步解开 token。
    - **检索流**：将草稿 KV 追加到 CPU 全 KV 缓存，用中心 [MASK] 查询执行跨层选择，逐步将 top-k 条目传输到 GPU 空闲池。
5.  块 N 去噪完成，最终 KV 缓存写回 GPU 和 CPU DRAM，替换草稿缓存。
6.  交换双缓冲池，块 N+1 开始无延迟的去噪。

## 3. 实验设计

## 6. 主要结论与发现

- **准确率**：HERALD 在 5-10% KV 预算下达到近无损准确率（与 Dense 相差 1-3 个百分点），略高于 MAGE（2.5-5% 预算），远优于 InfiniGen（即使 20% 预算仍显著下降）。

- **吞吐**：HERALD 在所有上下文长度上峰值吞吐最高；相比 Dense，8K 提升 1.34-1.40x，16K 提升 1.81-2.13x，32K 提升 1.92-2.47x。InfiniGen 和 MAGE 吞吐仅为 Dense 的 0.21-0.50x。
- **加速趋势**：HERALD 的优势随上下文增长而扩大，因为 KV 卸载缓解了长上下文下 GPU 内存压力，支持更大批处理。

## 7. 优点

- **创新性**：利用块 dLLM 特有的两个结构性质（中心 [MASK] 查询代表性、step-0 草稿可作语义代理）解决了精确稀疏选择在卸载场景下的核心矛盾，方法新颖且优雅。
- **系统设计**：双流管道、双缓冲、CPU-GPU 协同检索的层次化流水线充分重叠通信与计算，实现近乎零开销的 KV 检索。
- **实用性**：与主流推理框架（SGLang、FlashInfer）集成，实现仅 3.7K C++ + 9.6K Python 代码；在多种模型和任务上验证，加速比随上下文增长，符合长上下文趋势。
- **消融清晰**：逐步拆解每个组件贡献，证明单 [MASK] 查询、草稿重叠等设计的必要性。
- **公平比较**：所有方法使用相同底层框架和注意力内核，对比公平。

## 8. 不足与局限

- **实验覆盖**：
    - 仅评估了三种 dLLM（SDAR、TraDo、LLaDA 2.0），未覆盖其他块 dLLM 变体（如 Fast-dLLM v2）。

- **硬件依赖**：
    - 要求 CPU 具备足够计算能力（如 AVX-512），且需大容量 DRAM 托管完整 KV 缓存，可能不适用于低端 CPU。

- **偏差风险**：
    - 准确率实验仅用 512 token 输出（即 8 块），可能无法代表极长输出场景下的表现。
    - 性能实验固定去噪步数 T=20（LongBench 平均），但不同任务可能有不同步数分布，未动态适配。
- **应用限制**：
    - 方法专为块 dLLM 设计，不适用于自回归 LLM 或早期非块扩散模型（如连续扩散模型）。

- **未提及**：未报告统计显著性检验、置信区间，也未讨论缓存逐出策略对长对话的影响。
