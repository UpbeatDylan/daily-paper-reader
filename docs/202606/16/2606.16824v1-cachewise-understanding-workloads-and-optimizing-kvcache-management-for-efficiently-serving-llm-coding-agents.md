## 摘要
编码代理是一种快速增长的LLM应用，以长时间运行的闭环会话方式执行，其中LLM生成与外部工具调用交替进行。然而，与聊天工作负载不同，它们的服务行为尚未得到广泛研究。我们通过收集真实世界的编码助手轨迹数据集来填补这一空白。我们的分析表明，编码代理会话会重复重用大型前缀，并产生持续的KVCache压力，传统LLM服务策略对此处理不佳。基于我们的分析，我们提出了CacheWise，一种KVCache管理层，可改善编码代理工作负载的KVCache重用。CacheWise结合了前缀感知调度和基于工具调用元数据的轻量级预测驱动的重用感知驱逐。在vLLM中实现并基于收集的轨迹进行评估，CacheWise将KVCache驱逐次数减少多达2-2.6倍，并将整个代理会话完成时间改善多达3.5倍。

## Abstract
Coding agents are a fast-growing LLM application, executing as long-running closed-loop sessions in which LLM generations alternate with external tool calls. Yet, unlike chat workloads, their serving behavior has not been studied extensively. We address this gap by collecting a dataset of real-world coding assistant traces. Our analysis shows that coding agent sessions repeatedly reuse large prefixes and create sustained KVCache pressure that conventional LLM serving policies handle poorly.   Based on our analysis, we present CacheWise, a KVCache management layer that improves KVCache reuse for coding agent workloads. CacheWise combines prefix-aware scheduling with reuse-aware eviction guided by lightweight predictions from tool call metadata. Implemented in vLLM and evaluated on the collected traces, CacheWise reduces KVCache evictions by up to 2-2.6x and improves total agent session completion time by up to 3.5x.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究背景**：LLM（大型语言模型）应用正快速扩展，其中**编码代理（coding agents）**（如Claude Code、GitHub Copilot等）成为一种重要的新范式。与传统的对话式（chat）工作负载不同，编码代理以**长时间运行的闭环会话（closed-loop sessions）**方式执行：LLM生成与外部工具调用（如运行代码、查找文件、执行测试）交替进行，形成一个多轮交互的闭环。
- **核心问题**：现有LLM服务系统（如vLLM、Mooncake）主要针对对话场景设计，采用先来先服务（FCFS）批处理和最近最少使用（LRU）KVCache驱逐策略。这些策略**无法有效应对编码代理工作负载的独特特征**：
  - 会话会**重复重用大型前缀**（prefix），上下文长度随会话增长而急剧增加（可达数万token）。
  - 大量请求由**工具完成触发**，而非用户输入，导致请求间存在不可预测但具有结构性的时间间隔。
  - 因此，系统面临持续的**KVCache压力**，导致频繁的缓存驱逐和重计算，严重降低服务效率和用户感知的性能（会话完成时间）。
- **研究意义**：解决编码代理的服务效率问题对于实际部署至关重要，而此前缺乏公开的真实世界轨迹数据和分析。本文首次收集并公开了真实编码助手轨迹数据集（CATraces），并基于分析提出优化方案。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **整体思路**：提出**CacheWise**，一个**KVCache管理层**，在通用LLM服务系统（如vLLM）基础上增加**前缀感知调度**和**预测性KVCache驱逐**，以最大化缓存重用，减少驱逐和重计算开销。
- **核心技术与关键细节**：
  - **前缀感知请求调度**（Prefix-aware Request Scheduling）：
    - 调度时**优先选择与当前GPU内存中已存在KVCache前缀重叠度最高的请求**（即需要最少额外分配块`a_i(t)`的请求）。
    - 该策略直接减少了为满足新请求而驱逐其他会话缓存的可能性，隐式实现了类似**最短作业优先**（SJF）的效果，适合优化会话级完成时间而非单个请求延迟。
  - **预测性KVCache驱逐**（Predictive KVCache Eviction）：
    - 目标是**近似Belady最优策略**，即驱逐下一次被重用时间最远的块。
    - 关键洞察：工具调用元数据（工具名称、参数）蕴含了可预测的执行时间结构。例如，`ls`耗时短，`pytest`耗时长。通过**轻量级预测器**估计每个会话KVCache块的下次重用时刻 `E[τ_i(t)]`，然后**选择该值最大的块进行驱逐**。
    - 预测器训练：利用历史轨迹中**同一工具类型下按参数TF-IDF向量聚类**的分布，得到每个簇的工具执行时间分布。预测时根据当前工具调用的参数嵌入找到所属簇，从而估计剩余完成时间。
    - 实现细节：在vLLM的KVCache管理器中，为每个块关联会话元数据（工具名称、参数、调用时间）。当块变为未引用时，插入一个**驱逐堆**，按预测的下次重用时间排序。定期（每N次引擎迭代，实验中取N=3）重新计算堆中块的预测值以更新顺序。
- **工作流**：
  1. 请求到达后，负载均衡器根据**前缀匹配**路由到拥有最大可重用前缀的推理节点（利用已有路由机制）。
  2. 推理节点内的CacheWise调度器执行前缀感知调度。
  3. 当需要为新请求分配空间而容量不足时，执行预测性驱逐，从预测的下次重用时间最长的会话中驱逐块。
