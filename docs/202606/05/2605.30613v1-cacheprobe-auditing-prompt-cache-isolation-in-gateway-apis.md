## 摘要
在过去的一年中，大型语言模型（LLM）中的提示缓存在推理API中变得越来越流行。提示缓存通过重复使用特定提示的KV缓存的一部分来处理另一个请求，从而节省宝贵的计算资源并加快响应时间。然而，许多提示缓存的实现无法抵御时序攻击甚至基本的元数据泄露。Gu等人（ICML 2025）开发了一种审计LLM中提示缓存的方法。本文研究了OpenRouter的API网关架构是否引入了绕过提供商级别提示缓存隔离保证的提示缓存漏洞。大多数LLM推理提供商实现每账户或每组织的提示缓存以防止数据泄露，但通过共享组织凭证的路由经过OpenRouter是否会无意中导致所有OpenRouter用户之间的全局缓存共享？

## Abstract
Over the past year, prompt caching in Large Language Models (LLMs) has become increasingly more popular across inference APIs. Prompt caching helps save precious compute resources and speeds up response times by reusing parts of the KV cache of a specific prompt for another request. However, many implementations of prompt caching are not secure against timing attacks or even basic metadata disclosure. Gu et al. (ICML 2025) develop a method to audit prompt caching in LLMs. This paper investigates whether OpenRouter's API gateway architecture introduces prompt caching vulnerabilities that bypass provider-level prompt cache isolation guarantees. Most LLM inference providers implement per-account or per-organization prompt caching to prevent data leaks, but does routing through OpenRouter with shared organizational credentials inadvertently create global cache sharing across all OpenRouter users?

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：提示缓存（Prompt Caching）在LLM推理API中广泛应用，通过复用KV注意力状态来节省计算资源、降低延迟（GPU推理加速1.5–10倍，CPU推理加速20–70倍）。提供商通常提供缓存输入令牌折扣（如OpenAI自动90%折扣）。
- **已有安全问题**：Gu等人（ICML 2025）发现通过时序侧信道（TTFT测量）可以检测缓存是否存在，并逐步重建其他用户的提示。为此，大多数提供商实现了**每账户或每组织级别的缓存隔离**。
- **核心问题**：API网关（如OpenRouter）作为中间层，将所有用户请求通过**共享的组织凭证**转发给上游提供商。这种架构是否无意中**绕过了提供商级别的缓存隔离**，使得所有OpenRouter用户共享全局缓存，从而暴露跨用户提示内容？
- **研究意义**：揭示网络中间件引入的安全漏洞，为API网关设计和缓存隔离策略提供重要参考。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：利用两种信号检测缓存共享——**时序侧信道分析**（缓存命中时TTFT显著降低）和**元数据泄露**（部分API响应中返回`cached_tokens`字段）。
- **关键技术细节**：
  - **攻击场景**：攻击者持有有效的OpenRouter API密钥（含信用额度），可向目标模型发送任意提示并测量TTFT。通过OpenRouter的`provider.order`参数强制路由到特定提供商。
  - **实验流程（交错采样法）**：
    1. 发送唯一随机提示（建立缓存未命中基线）；
    2. 模拟受害者填充缓存（发送不同随机提示）；
    3. 发送前缀匹配的探测提示（测试缓存命中）。
    通过交错“未命中”和“命中”测试，消除网络延迟和API负载波动的影响。
  - **提示设计**：使用4,096个令牌的提示，95%前缀相同（前3,891个令牌相同，后205个不同），模拟攻击者部分前缀重建。
  - **间隔**：请求间插入500ms延迟，保障缓存条目持久化，同时避免速率限制。
  - **统计检测**：使用双样本Kolmogorov-Smirnov检验比较命中/未命中TTFT分布，显著性阈值设为`p < 10^(-8)`。元数据检测以超过50%的缓存请求报告`cached_tokens`超过预期前缀长度90%为阳性。
- **6种测试场景**：直接API同账户/跨账户；OpenRouter默认模式同账户/跨账户；OpenRouter BYOK（自带密钥）模式同账户/跨账户。

  - 场景3-4：OpenRouter默认模式（同账户/跨账户）——测试共享凭证影响。
  - 场景5-6：OpenRouter BYOK模式（同账户/跨账户）——验证恢复隔离。
- **提供商与模型**：

- **基准**：以直接API跨账户（应无缓存共享）作为“正确隔离”基准，以直接API同账户（应有缓存）作为“缓存功能正常”基准。

## 6. 论文的主要结论与发现

- **直接API基线**：三个提供商均正确实现跨账户缓存隔离（同账户可检测缓存，跨账户无缓存命中）。
- **OpenRouter默认模式**：**所有三个提供商均表现出跨用户缓存共享**。
  - **Groq**：时序统计虽不显著（p=0.002），但元数据揭示100%缓存命中率（跨账户）。
  - **Fireworks**：时序统计极其显著（跨账户p=4.08e-15），为最强证据。
  - **OpenAI**：元数据显示4.8%的跨账户请求命中缓存（低比率但存在）。
- **OpenRouter BYOK模式**：跨账户测试无任何缓存信号（p>0.16，无元数据），**恢复隔离**。
- **根本原因**：漏洞源于OpenRouter使用**共享组织凭证**，而非路由层本身；提供商将OpenRouter所有流量视为同一组织，导致缓存池化。
- **攻击可操作性**：任何OpenRouter用户均可主动探测缓存，逐步重建其他用户的提示（包括系统提示、RAG检索模式、敏感数据等）。

## 7. 优点：方法或实验设计上的亮点

1. **系统化的测试矩阵**：通过6种场景交叉对比，精确定位漏洞来源为共享组织凭证，而非路由或网络因素。
2. **双重检测机制**：同时使用时序分析（统计检验）和元数据披露，提高检测可靠性和解释性。
3. **控制变量充分**：采用交错采样法消除时段波动，使用公共云VM减少网络差异，设置严格统计阈值（p<10^(-8)）降低误报。
4. **负责任披露**：2025年11月14日首次联系OpenRouter，12月12日发送论文草稿，体现学术伦理。
5. **结果可复现**：提供完整源代码、原始时序数据和配置（GitHub仓库公开）。

## 8. 不足与局限

1. **提供商和模型覆盖有限**：仅测试3个提供商各1个模型，可能不适用于其他提供商（如Anthropic、Mistral）或模型。
2. **提示类型单一**：使用随机ASCII字母序列，而非真实应用中的自然语言提示；真实缓存命中模式可能不同（如用户重用系统提示）。
3. **统计阈值保守**：p<10^(-8)可能漏掉部分或间歇性的缓存泄露（但论文回应“保守但必要”）。
4. **未考虑提供商端缓存策略变化**：实验中假设缓存持久化（500ms间隔），但未测试更长时间间隔或多跳缓存。
5. **未评估攻击实际影响**：仅证明存在缓存共享，未演示完整提示重建（如Gu等人的逐令牌重建方法）。
6. **OpenRouter响应不完整**：OpenRouter团队虽已联系但未给予实质回应，未进行联合缓解测试。
7. **BYOK模式局限性**：虽能恢复隔离，但牺牲了OpenRouter的便利性和定价优势，并非根本解决方案。
