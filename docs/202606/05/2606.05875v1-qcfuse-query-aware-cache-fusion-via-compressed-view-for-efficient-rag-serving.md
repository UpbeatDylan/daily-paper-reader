## 摘要
检索增强生成（RAG）通过将生成过程锚定在外部证据上，提高了大语言模型（LLM）的答案质量，但处理检索到的上下文使得预填充阶段成为主要的服务成本。RAG缓存融合通过复用检索到的块的预计算键值（KV）缓存，并在当前提示下选择性重计算令牌来降低这一成本。然而，现有的选择器面临着质量和效率之间的两难困境：快速的查询无关或最终层查询到上下文选择器可能遗漏与请求相关的证据，而全视图查询感知选择器在重计算之前需要广泛的上下文和层可见性，从而阻碍了逐层缓存融合流水线。我们提出了QCFuse，一种用于RAG缓存融合的压缩视图查询感知选择器。QCFuse使用块锚点查询探测来将用户查询状态条件化在紧凑的每块锚点上，并通过关键层概览来识别重计算令牌而无需检查所有层。我们在SGLang中实现了QCFuse，并在六个数据集上使用四个开放权重LLM进行了评估。QCFuse达到了完全预填充级别的质量。在匹配质量下，QCFuse相对于完全预填充实现了平均1.7倍的预填充时间加速，相对于最强的质量保持基线ProphetKV实现了1.5倍的加速。

## Abstract
Retrieval-augmented generation (RAG) improves large language model (LLM) answer quality by grounding generation in external evidence, but processing retrieved contexts makes the prefill stage a dominant serving cost. RAG cache fusion reduces this cost by reusing precomputed key-value (KV) caches for retrieved chunks and selectively recomputing tokens under the current prompt. Existing selectors, however, face a dilemma between quality and efficiency: fast query-agnostic or final-layer query-to-context selectors can miss request-relevant evidence, whereas full-view query-aware selectors require broad context and layer visibility before recomputation and therefore stall the layer-wise cache-fusion pipeline. We present QCFuse, a compressed-view query-aware selector for RAG cache fusion. QCFuse uses chunk-anchor query probing to condition user-query states on compact per-chunk anchors and critical-layer profiling to identify recomputation tokens without all-layer inspection. We implement QCFuse in SGLang and evaluate it on four open-weight LLMs across six datasets. QCFuse reaches full-prefill-level quality. At matched quality, QCFuse achieves an average prefill-time speedup of 1.7x over full prefill and 1.5x over ProphetKV, the strongest quality-preserving baseline.

## 1. 论文的核心问题与整体含义（研究动机和背景）

检索增强生成（RAG）通过融合外部知识库检索到的证据来提升大语言模型（LLM）的生成质量。然而，在RAG服务中，预填充（prefill）阶段需要处理大量检索到的上下文（chunks），成为主要的延迟开销。缓存融合（Cache Fusion）方法通过预计算并复用这些chunks的键值（KV）缓存，仅选择性地重计算少量token，以加速预填充。然而，现有选择器面临 **质量与效率的两难**：查询无关选择器（如CacheBlend、EPIC）或仅使用最终层注意力信号的选择器（如FusionRAG）速度快但可能遗漏请求相关证据；而全视图查询感知选择器（如ProphetKV）需加载完整上下文KV缓存并扫描所有层，虽质量高但会阻塞逐层流水线，导致选择阶段成为瓶颈。

**核心问题**：如何设计一个既保持查询感知（query-aware）选择质量，又能兼容流水线并行、避免预填充KV缓存加载阻塞的选择器？

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想
QCFuse提出**压缩视图查询感知选择器**，通过两种压缩策略在不牺牲质量的前提下降低选择阶段的KV缓存加载量：
- **token-view压缩**：使用**chunk-anchor query probing**，即离线为每个chunk选取代表性锚点token（如KVzip），在线时仅用锚点KV缓存来条件化用户查询状态，避免加载完整上下文。
- **layer-view压缩**：通过**critical-layer profiling**离线识别对token定位最关键的少数层（top-3），在线时仅使用这些层的注意力信号进行token重要性评分，避免扫描所有层。

### 2.2 关键技术细节

#### 阶段I：离线缓存准备
- **PIC缓存**：为每个可重用chunk独立预计算并存储KV缓存（位置无关缓存）。
- **锚点选择**：对每个chunk，使用KVzip规则（自监督重复上下文重构探针）或Knorm、Sink等规则，保留10%的token作为锚点。
- **关键层剖析**：在SQuAD、NewsQA、Natural Questions等带有明确答案标签的数据集上，通过Recall@10%度量单层定位能力，选出top-3关键层（通常位于模型中间层）。

#### 阶段II：在线查询感知融合
1. **锚点条件化查询探针**：加载每个检索chunk的锚点KV缓存，拼接成紧凑上下文𝐶_A；让用户查询token对其进行注意力计算，得到上下文条件化的查询状态。
2. **关键层token评分**：仅使用profiled关键层的注意力信号（查询状态对完整上下文键的注意力），计算每个上下文token的重要性分数，选出top-ρ% token作为重计算集合P。
3. **逐层流水线缓存融合**：基于P，使用Triton优化内核仅对选定token进行稀疏重计算，同时与后续层的KV缓存加载重叠。

### 2.3 公式流程
- 离线：每chunk锚点集 A_i = TopK(G_i, r_a|c_i|)；关键层集 L^* = TopK(R(l), k)（k=3）。
- 在线：查询状态 Q^l_U(𝐶_A) ← LLM^l( U; KV^A(𝐶_A) )；重要性 I_{L^*}(t) = Σ_{l∈L^*} Attn^l(Q^l_U(𝐶_A), K^{PIC,l}_C)_t；P = TopK(I(t), ρN)。

## 3. 实验设计

### 3.1 数据集与场景
- **LongBench**（自然语义QA）：MuSiQue（多跳）、2WikiMQA（多跳）、HotpotQA（多跳）。
- **RULER**（合成长上下文任务）：多查询检索（MQ）、多值检索（MV）、变量追踪（VT）。
- 每个数据集200个样本，上下文由20个512-token chunks组成。

### 3.2 基准方法
- **Full prefill**（完全预填充，质量上限）
- **Direct PIC reuse**（纯复用，无重计算）
- **CacheBlend**（查询无关，基于KV偏差）
- **EPIC**（查询无关，静态边界token）
- **FusionRAG**（仅最终层查询感知）
- **ProphetKV**（全视图查询感知，最强质量保持基线）

所有方法使用相同检索chunks、重计算比率ρ（0.1~0.5）、缓存布局和逐层重计算流水线。

### 3.3 评估指标
- **任务质量**：LongBench采用F1分数，RULER采用字符串匹配（SM），并归一化为百分比。
- **TTFT**（首次token生成时间，包含选择、缓存加载、重计算、生成）。
- **服务吞吐量**：请求/秒 vs TTFT曲线。

- **质量恢复**：QCFuse在0.5重计算比率下达到或超过完全预填充质量，甚至在某些语义QA任务上略高于全预填充（可能因去除噪声）。

- **关键层有效性**：Top-3 profiled中间层提供了最佳token定位精度，优于首层、末层或全层聚合（图16）。
