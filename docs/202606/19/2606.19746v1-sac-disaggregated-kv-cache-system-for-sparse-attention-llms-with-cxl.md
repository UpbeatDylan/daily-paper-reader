## 摘要
大语言模型向长上下文推理的扩展已将主要服务系统的瓶颈从计算转变为内存容量。传统的密集注意力模型解决方案依赖基于RDMA的解耦内存池，在解码前将整个前缀KV缓存从远程存储粗粒度地获取到本地内存。然而，这种方法对于新兴的稀疏注意力模型而言效率低下。解码期间仅有一小部分KV条目活跃，但这些系统仍然会本地获取完整KV缓存，导致严重的传输瓶颈和本地内存浪费。为解决此问题，我们提出SAC，这是首个针对稀疏注意力模型优化的高效解耦KV缓存系统。通过利用Compute Express Link（CXL）的低延迟、缓存行粒度的加载/存储语义，SAC在推理期间仅按需获取所需的top-k KV条目。使用SGLang在DeepSeek-V3.2上的评估表明，与基于RDMA的基线相比，SAC实现了2.1倍的吞吐量提升、9.7倍的TTFT降低和1.8倍的TBT降低，将基于CXL的解耦确立为新兴稀疏注意力模型的优越基础设施。

## Abstract
The scaling of LLMs toward long-context inference has shifted the primary serving system bottleneck from computation to memory capacity. Traditional solutions for dense attention models rely on RDMA-based disaggregated memory pools, which perform coarse-grained fetching of the entire prefix KV cache from remote storage to local memory before decoding. However, this approach is fundamentally inefficient for emerging sparse attention models. While only a small fraction of KV entries are active during decoding, these systems still fetch the full KV cache locally, leading to severe transmission bottlenecks and local memory wastage. To address this, we propose SAC, the first efficient disaggregated KV cache system optimized for sparse attention models. By leveraging the low-latency, cache-line granularity load/store semantics of Compute Express Link (CXL), SAC fetches only the required top-k KV entries on demand during inference. Evaluations on DeepSeek-V3.2 using SGLang show that SAC achieves 2.1x higher throughput, 9.7x lower TTFT, and 1.8x lower TBT compared to RDMA-based baselines, establishing CXL-based disaggregation as the superior infrastructure for emerging sparse attention models.

## 1. 核心问题与整体含义（研究动机和背景）

随着大语言模型（LLM）向长上下文推理演进，服务系统的瓶颈从计算转变为内存容量，尤其是前缀 KV 缓存的管理成为关键挑战。传统方案依赖 RDMA 解耦内存池，在解码前**预取整个前缀 KV 缓存**到本地内存。然而，对于新兴的**稀疏注意力模型**（如 DeepSeek-V3.2、GLM-5.1），解码时仅有少量 KV 条目（top-k）活跃，但全量预取策略仍会将完整的 KV 缓存全部拉取到本地，导致两个根本问题：
- **传输瓶颈**：大量 KV 缓存传输占用 RDMA 带宽，高并发下产生严重排队延迟，降低首词时间（TTFT）和整体吞吐。
- **本地内存浪费**：只使用极少部分 KV 条目，但全量缓存需驻留本地，导致 TB 级内存需求，成本高昂且效率低下。

因此，需要一种能**按需只获取活跃 top-k KV 条目**的新架构，而 RDMA 因高延迟和协议开销无法胜任。论文提出利用 **Compute Express Link（CXL）** 的近内存延迟、缓存行粒度加载/存储语义，构建首个针对稀疏注意力模型优化的解耦 KV 缓存系统 **SAC**。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
将 KV 缓存完整存放在 CXL 解耦内存池中，解码时**实时按需读取每层 top-k KV 条目**，避免全量预取。利用 CXL 接近 DRAM 的延迟和零额外协议开销，使得细粒度、非连续的小数据块读取可行。

### 系统架构（三个组件）
1. **Prefill 实例**：处理预填充阶段（计算密集），计算完整 KV 缓存并写入 CXL 内存池。
2. **Decode 实例**：基于 HiSparse 框架（SGLang 的稀疏推理扩展），在每层注意力计算时，动态选择 top‑k 索引，从 CXL 内存池**直接读取**所需 KV 向量到 GPU HBM，并写回新生成的 KV。
3. **CXL 解耦内存池**：由多个 CXL 设备通过 CXL 交换机互联，统一管理 KV 缓存及其元数据，提供全局地址空间。

### 关键技术细节
- **内存布局**：采用“层优先”（layer-first）布局，与本地 GPU 一致，避免 RDMA 所需的布局转换。
- **元数据管理**：将索引存于 CXL 共享内存区域，用原生 load/store 操作代替 RPC，减少延迟。
- **读写优化**：使用向量化加载/存储指令（`ld.global.b64` / `st.global.b64`），通过 CUDA 内核实现内存合并访问。
- **设备交错策略**：将单个请求的 KV 缓存分配在同一 CXL 设备上，调度时按轮询方式将请求分配给不同 GPU 对应的 CXL 设备，缓解多 GPU 同时访问同一设备的带宽争用（图 8）。
- **与 HiSparse 集成**：设置 `device_buffer_size` 参数（实验中默认 6144），缓存热 KV 条目以减少 CXL 读取次数。

## 3. 实验设计

### 数据集与场景
- **数据集**：ShareGPT 数据集（512 条请求），固定输出长度 1K tokens（主实验），额外实验覆盖 2K/4K/8K output。
- **上下文长度**：16K、32K、64K、128K tokens。
- **场景**：
  - **Round-1（Cache Populate）**：模拟预填充阶段，计算并写入 KV 缓存到内存池。
  - **Round-2（Cache Hit）**：模拟解码阶段，KV 缓存已预存，仅从池中读取 top‑k 条目。
- **并发规模**：Round-1 并发 8，Round-2 并发 64（主实验）；可扩展性实验中并发从 8 到 192 不等。

### 对比方法
- **RDMA 内存池**：基于 ConnectX-7 100Gbps NIC 的环回配置（最佳情况基线）。
- **本地 DRAM**：非解耦的本地主机内存（上界基线）。
- **额外对比**：GPU-only HBM（全驻留）。

### 评估指标

## 6. 主要结论与发现

- **SAC 显著优于 RDMA 基线**：在 Round-2（解码阶段），SAC 平均吞吐提升 **2.1×**，TTFT 降低 **9.7×**，TBT 降低 **1.8×**。
- **接近本地 DRAM 性能**：SAC 解码吞吐达到本地 DRAM 的 **91%**，仅衰退 9%。
- **可扩展性强**：随着并发度增加，SAC 吞吐持续线性增长，而 RDMA 因传输瓶颈迅速饱和。在 128K 上下文时，SAC 吞吐可达 RDMA 的 **3.1×**。
- **CXL 设备交错有效**：双设备交错平均提升吞吐 9.2%，峰值 14.2%。
- **HiSparse 缓冲区增大有益**：6K 缓冲区比 4K 提升吞吐 10.4%，降低 CXL 读取压力。
- **CXL 是稀疏注意力 LLM 的优越解耦基础设施**，解决了 RDMA 固有的低效问题。

## 7. 优点

1. **问题识别精准**：首次系统分析 RDMA 全量预取在稀疏注意力中的两大瓶颈（传输瓶颈、内存浪费）。

3. **端到端系统实现**：集成到 SGLang/HiSparse 框架，提供完整工作流（预填充、解码、元数据管理、设备交错），实用性强。
4. **实验设计严谨**：包含多种上下文长度、输出长度、并发度，对比多个基线，并做消融与尾延迟分析，统计稳定性良好。

6. **可扩展性考量**：提出设备交错策略缓解多 GPU 争用，并讨论了未来模型（DeepSeek-V4）的通用性。

## 8. 不足与局限

1. **模型覆盖有限**：仅评估 DeepSeek-V3.2，其他稀疏模型（如 GLM-5.1、DeepSeek-V4）未验证，通用性待确认。
