## 摘要
长上下文大语言模型（LLM）推理日益受到键值（KV）缓存的内存占用和解码成本的制约，限制了在资源受限硬件上的可持续部署。现有的KV缓存驱逐方法通常对基于GQA的LLM中的所有注意力头应用启发式令牌评分。这些方法忽略了注意力头的不同功能，导致关键令牌被驱逐，从而降低了LLM的性能。为解决这一问题，我们提出了CompressKV，一种面向基于GQA的LLM的资源高效KV缓存压缩框架。CompressKV不聚合所有头的注意力分数，而是识别语义检索头（SRH），这些头捕捉提示词的初始和最终令牌以及语义重要的中间上下文证据，并利用它们来选择应保留KV对的令牌。此外，CompressKV根据层间驱逐误差的离线估计在各层之间分配缓存预算。在LongBench和Needle-in-a-Haystack上的实验表明，CompressKV在各种内存预算下均一致优于现有的KV缓存驱逐方法。值得注意的是，在LongBench问答任务中，它仅使用3%的KV缓存就保留了超过97%的全缓存性能，并在Needle-in-a-Haystack上仅使用0.7%的KV存储就达到了90%的准确率。这些结果展示了长上下文LLM推理中改进的资源-性能权衡。我们的代码公开于：https://github.com/TUDa-HWAI/CompressKV

## Abstract
Long-context large language model (LLM) inference is increasingly constrained by the memory footprint and decoding cost of key-value (KV) caches, limiting sustainable deployment on resource-constrained hardware. Existing KV cache eviction methods typically apply heuristic token scoring over all heads in GQA-based LLMs. These methods ignore the different functionalities of attention heads, leading to the eviction of critical tokens and thus degrading the performance of LLMs. To address this issue, we propose CompressKV, a resource-efficient KV-cache compression framework for GQA-based LLMs. Instead of aggregating attention scores from all heads, CompressKV identifies Semantic Retrieval Heads (SRHs) that capture both the initial and final tokens of a prompt and semantically important mid-context evidence, and uses them to select tokens whose KV pairs should be retained. Furthermore, CompressKV allocates cache budgets across layers according to offline estimates of layer-wise eviction error. Experiments on LongBench and Needle-in-a-Haystack show that CompressKV consistently outperforms existing KV-cache eviction methods across memory budgets. Notably, it preserves over 97\% of full-cache performance using only 3\% of the KV cache on LongBench question-answering tasks and achieves 90\% accuracy with just 0.7\% KV storage on Needle-in-a-Haystack. These results demonstrate an improved resource--performance trade-off for long-context LLM inference. Our code is publicly available at: https://github.com/TUDa-HWAI/CompressKV

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：长上下文大语言模型（LLM）推理中，键值（KV）缓存的内存占用和解码成本随序列长度线性增长，成为资源受限硬件上可持续部署的主要瓶颈。
- **现有方法局限**：基于分组查询注意力（GQA）的LLM中，现有KV缓存驱逐方法（如SnapKV、StreamingLLM等）对所有注意力头采用统一评分或简单聚合，忽略不同注意力头的功能异质性。尤其是“流式头”（Streaming Heads）倾向于关注提示词的起始和结尾，导致中间关键上下文token被错误驱逐，损害模型性能。
- **整体含义**：需要一种能够感知注意力头功能差异、利用“检索型”头保留语义重要token的压缩方法，以在极低内存预算下维持推理性能。

## 2. 方法论

### 核心思想
- 识别每层中能够捕捉初始/最终token以及语义重要的中间上下文的注意力头——**语义检索头（Semantic Retrieval Heads, SRH）**。
- 仅利用这些SRH的注意力分数来选取需保留KV对的token，避免流式头的霸权。
- 离线估计每层压缩误差，按比例分配缓存预算，实现层自适应分配。

### 关键技术细节
1. **SRH识别标准**（公式1）：
   - 使用校准数据集，对每个头 \( h \)，聚合其在生成正确答案span时的注意力质量：
     \[
     S_{\text{SRH}}(h) = \sum_{t=1}^N \mathbf{1}\{y_t \in \mathcal{A}\} \sum_{j \in \mathcal{A}} a_{t,j}^{(h)}
     \]
     - \( y_t \)：第 \( t \) 步生成token；\( \mathcal{A} \)：正确答案span；\( a_{t,j}^{(h)} \)：头 \( h \) 对第 \( j \) 个token的注意力权重。
   - 相比传统检索头（TRH）的严格top-1标准，SRH通过聚合整个答案span的注意力，能识别出虽未达到单token峰值但关注语义邻域的头。

2. **令牌选择驱动**：
   - 每层选取得分最高的 \( k \) 个SRH（实验中取 \( k=4 \)）。
   - 将这些头的注意力分数矩阵在观察窗口内求和池化，再平均后取top-\( N \) 个token，其KV对被保留，其余丢弃。
   - 同一层内所有头共享该token索引集合。

3. **层自适应缓存分配**（基于误差感知）：
   - 在解码阶段，以全缓存输出为参照，计算每层压缩后的注意力块输出相对误差：
     \[
     e^{(l)} = \sum_{t=1}^T \frac{\|\Delta O_t^{(l)}\|_F}{\|O_{t,\text{full}}^{(l)}\|_F + \epsilon}, \quad \tilde{e}^{(l)} = \frac{e^{(l)}}{\sum_k e^{(k)}}
     \]
   - 给定总预算，先为每层分配最小/最大限额（\( m=32, M=3\times B_{\text{per-layer}} \)），剩余按归一化误差比例分配。
   - 该离线分析仅执行一次，推理时复用，不增加在线开销。

## 7. 优点

- **头部功能感知**：创新地利用语义检索头而非聚合所有头，缓解了流式头主导导致关键token被驱逐的问题，尤其适合GQA模型。
- **离线预计算**：SRH识别和层误差预算分配均离线完成，推理时零额外开销，实用性强。
- **极低预算下的鲁棒性**：在128–256极低预算下优势显著，远超基线。
- **通用性**：在多个模型族（Llama, Mistral, Qwen）和规模（7B–32B）上验证有效。
- **可组合性**：与主流加速（MInference, XAttention）和量化（KIVI）方法正交，易于集成。

## 8. 不足与局限

- **模型范围有限**：仅测试了GQA结构模型（Llama-3.1, Mistral, Qwen2.5），未覆盖MHA或其他注意力机制（如MLA），结论能否推广尚不确定。
- **校准数据集依赖**：SRH识别需要一个校准数据集（沿用[27]），该数据集的选择可能影响头部分配的泛化性，未讨论跨领域鲁棒性。
- **静态分配**：层预算和SRH集在推理前固定，无法根据实际输入动态调整，可能不适应分布外样本。
- **未报告计算开销细节**：离线分析的耗时和内存未量化，难以评估完整部署代价。
- **未涉及安全性评估**：未讨论压缩是否引入额外幻觉或偏见风险。
