# SM ETRIC : Rethink LLM Scheduling for Serving Agents with Balanced Session-centric Scheduling 

## Jiahao Wang 1, Kaizhan Lin †1 , Kaixi Zhang 1, Jinbo Han 1, Xingda Wei 1, Sijie Shen 2,Chenguang Fang 2, Wenyuan Yu 2, Rong Chen 1, Haibo Chen 11Institute of Parallel and Distributed Systems, Shanghai Jiao Tong University 2Alibaba Group 

# Abstract 

LLM scheduling is critical to serving, yet it remains unclear how well existing designs fit agentic serving—with LLM re-quests issued by agents instead of humans. This shifts the workload in two ways: (1) agents act only on complete re-sponses, making the cluster’s tokens per second (TPS) the primary goal and relaxing—not eliminating—per-token la-tency requirements; and (2) requests share much of their KV$—reuse exceeds 80 % of request tokens in a production trace from B AILIAN , versus 54–62 % in chat. This paper first contributes a systematic study of request scheduling for agents on two real-world traces. We find that to increase KV$ reuse, existing schedulers overly prioritize routing requests to instances caching their KV$, overload-ing a few while leaving the rest idle, capping TPS. We thus present two key insights: (1) load balance need not sacrifice all KV$ reuse, thanks to the global-tier KV$ store and (2) by utilizing the workload’s intra-session locality, balancing a small fraction of requests—the first request in each agent session—suffices to balance the cluster without sacrificing most KV$ reuse on local instances. SM ETRIC realizes these insights with balanced session-centric scheduling: it routes each session’s first request purely for load balance and its follow-up requests in a cache-aware manner, preserving load balance and local reuse while keep-ing demand on the global tier low. Using the session turn information as the scheduling metric is deliberate: it is de-rived efficiently and accurately from the user inputs alone, so the scheduler stays clean and stateless. SM ETRIC improves cluster TPS by 10–16 % under prefill-decode colocation with a global store and prefill TPS by 2–34 % under disaggregation over state-of-the-art schedulers, also with a better per-token latency.

Large language models (LLMs) now power agents like Claude Code [ 6 ] and OpenClaw [ 36 ], which accomplish complex tasks such as writing [ 5], debugging [ 14 ], and tuning the

Therefore, how efficiently a model provider serves these tokens determines both the speed and the cost of running agents. To that end, existing providers deploy clus-ters dedicated to serving requests from many users’ sessions, each cluster running many instances, each able to serve a complete request. We term this cluster-scale, agent-dedicated LLM serving agentic serving in this paper. Note that agentic serving is still LLM serving: the provider serves each request through the standard LLM API and neither runs the agents nor sees their internal logic. What changes is the workload, because the requests now come from agents. Request scheduling—how the cluster routes each request to an instance—is a key pillar of LLM serving. Though well-studied recently [ 54 , 59 , 42 , 53 , 17 , 2], we argue that current scheduling designs must be revisited for agentic serving, for two reasons: First, agentic serving requires a different performance ob-jective: the cluster’s tokens per second (TPS) becomes the primary goal, while the per-token latencies targeted by prior work [ 54 , 59 , 58 ]—time-to-first-token (TTFT) and time-per-output-token (TPOT)—still cannot be sacrificed, only relaxed. This shift is driven by two properties of agentic workloads. (1) An agent emits far more tokens per request than a human chat user does [ 9 ], so a cluster must sustain high TPS to serve the load at low cost. Besides cost, a cluster whose TPS falls short of demand queues requests, so every agent waits longer for its response. (2) An agent tolerates delays on any individ-ual token—the first or a later one—as long as the complete response arrives quickly: unlike a human who reads tokens as they stream [ 8], an agent acts only on the complete response, e.g., to decide which tool to call. Latency requirements still hold, however: a response that is too slow makes the agent treat the request as timed out [ 7]; these requirements are only more relaxed than for humans. Second, agentic serving exhibits a different pattern from chat: KV$ reuse dominates. Specifically, the efficiency of serving an LLM request depends heavily on KV$ reuse, where the KV$ is the intermediate state a cluster caches to avoid recomputation (§2). In chat workloads, this reuse is modest: only 54–62 % of request tokens are reused from cache [ 45 ]. In 1arXiv:2607.08565v1 [cs.DC] 9 Jul 2026 agentic serving, KV$ reuse dominates: by analyzing the pro-duction trace from a cluster at one of the world’s largest token providers (B AILIAN ), we find reuse exceeds 80 % (Find-ing 1). Throughout, we mark such key empirical observations as numbered Findings , all presented in §3. This shift requires revisiting whether current scheduling designs remain effective when KV$ reuse dominates at this scale. In this paper, we conduct the first systematic study of re-quest scheduling for agents, using two recent large-scale real-world agentic serving traces (§3). By studying both state-of-the-art schedulers and the production scheduler of B AILIAN ,we find that the key issue is not that they fail to reuse the KV$, but that they overly prioritize it: they route each request to the instance holding its cached KV$ (cache-aware), trading load balance for reuse. In particular, first-turn requests across sessions of the same agent framework share similar system prompts (Finding 3), so a cache-aware scheduler routes them all to the few instances caching these prompts. Moreover, since each follow-up request is likely routed to the instance caching its session’s earlier turns, every session is pinned to the instance where its first request lands; these few instances thus absorb entire sessions and become overloaded, while the rest of the cluster sits underutilized. Sacrificing load balance is harmful: an imbalanced cluster leaves the compute of underloaded instances idle, directly capping TPS. Worse, the sacrifice is unnecessary, as we find the token computation of real-world agentic workloads can 

be balanced across instances at scale (Finding 6). Meanwhile, simply abandoning cache-awareness for balance is no better, since reuse saves the computation of most prefill tokens, as discussed above. To address this dilemma, we draw two insights. First, from the system’s perspective, pursuing load balance does not have to compromise KV$ reuse: modern agentic serving adopts a two-tier KV$ store—a local tier on each instance’s GPU memory and a global tier on cluster-wide CPU memory (§2). Even if the scheduled instance does not hold the KV$ locally, it can fetch it from the global tier. As long as the global tier serves these KV$ fetches in time, TPS is unaffected (§4.1); the only cost is a small increase in TTFT, which agents tolerate as discussed above. The global tier, however, is not a free lunch: its capacity and bandwidth are limited, and routing 

every request purely for balance—funneling all reuse through global fetches—risks exceeding what it can hold and serve in time. The scheduler should therefore still exploit local-tier reuse. Second, from the agent workload’s perspective, once the scheduler considers local-tier KV$ reuse—i.e., prioritizes routing targets with high reuse, the common practice—the placement of each session’s first request largely determines the cluster’s load balance. This is because of the workload’s intra-session locality: (1) follow-up requests mostly reuse the KV$ of earlier turns in the same session (Finding 2), and (2) such reuse happens quickly (Finding 4), so when a follow-up request arrives, its KV$ typically still resides in the local tier of the instance that served the previous turn. Consequently, all requests of a session are likely routed to the same instance as its first request (§4.2). Scheduling first requests alone for load balance thus suffices to balance the cluster; and since the first requests of sessions are only a small fraction of all requests, the global tier alone is sufficient to support their KV$ reuse. Based on these two insights, we propose SM ETRIC : bal-anced session-centric scheduling for high-TPS agentic serv-ing. SM ETRIC incorporates agent session information into scheduling with a differential policy: for the first request of each session, it schedules purely for load balance, spread-ing sessions across all instances of the cluster; for follow-up requests, it follows traditional cache-aware routing that prior-itizes local-tier reuse, keeping demand on the global tier low. Using the session-turn information as a routing hint is our explicit design choice: this hint can be obtained in a router-stateless way from an existing LLM call, i.e., without the router keeping any record of users’ past requests. The router thus routes each request as a pure function of the request itself, keeping no per-session state to track or to reclaim as sessions go idle, which keeps the router simple and scalable. This simple policy also achieves balanced execution while preserving KV$ reuse, even for the first requests: their reuse comes mostly from the shared system prompt of the agent framework (Finding 3), which the scheduled instance can fetch from the global tier wherever the request lands. Finally, to handle the rare tail case where many long sessions accu-mulate on the same instance, SM ETRIC migrates sessions across instances upon imbalance. We have implemented SM ETRIC on vLLM [ 44 ] and LM-Cache [ 29 ], and extensively evaluated it against state-of-the-art schedulers, including the production scheduler used in B AILIAN , with serving paradigms spanning both prefill-decode colocation and prefill-decode disaggregation. # 2 Background and System Setup 

LLM and KV$. An LLM serves a request in two phases. Given the input tokens (the prompt ), a prefill pass pro-cesses the prompt with a single model forward (possibly split into chunks [ 1]) and emits the first output token. A

decode pass then emits the remaining tokens one by one in an auto-regressive manner, until the model produces an end-of-sequence (EOS) token. Each token comes from a single model forward whose input is the prompt followed by every token emitted so far, appended in generation order. The forward pass of LLMs requires computing a key (K) and a value (V) tensor for each token in the input sequence. Since these tensors are the same for inputs with the same prefix, it is standard to cache the KV tensors generated so far (the KV$) to accelerate the decode phase, as well as the prefill phase of later requests with the same prefix. As shown in Figure 4 (a), if a 3-token input request has two KV$ hits, then it only needs to compute the KV$ for one token. 

KV$ store hierarchy and the local vs. global tiers. Since cross-request reuse is common in production workloads [ 45 ,38 ], serving systems typically cache the KV$ of finished requests for some time. Caching requires storage capacity because each request’s KV$ is large: for example, a single 128K-token request consumes about 32 GB of KV$ on the popular Qwen3-32B model. As a result, the KV$ store in production systems follows a multi-tier architecture as shown in Figure 1. The first tier is device (GPU) memory; when it fills, KV$ cascades to the larger, global tiers below—host (CPU) memory, then cloud storage (e.g., S3). A later KV$ hit on a lower tier must load the entry back up to the GPU. Note that thanks to recent techniques including GPU-direct RDMA, the global tier can also fetch data from remote GPU memory ( ➂). In this paper, we consider only two tiers: the GPU memory tier (local tier) and a CPU memory tier linked by RDMA (global tier). This does not lose generality: the storage tier shares the same PCIe and RDMA link as the computation tier [ 47 ], so modeling it separately would not increase the speed of the global tier.

Finding 1: KV$ reuse is dominant: the reuse ratio ex-ceeds 80 % of the KV$ in both traces (Figure 5). Figure 5 profiles the workload KV$ reuse ratios in agentic serving scenarios: we can skip 80 % of the KV$ computation by reusing those generated so far, assuming all KV$ can be stored. We measure reuse at KV-cache block granularity using an idealized infinite KV store, where each block holds a fixed number of tokens, with the block size matching that used by the cluster serving the workloads. During prefill, each KV block is looked up in the store; hit blocks contribute the tokens whose computation can be skipped. # 4 Balanced session-centric scheduling 

4.1 Analysis of existing schedulers A preliminary on the LLM scheduling in a cluster. To the best of our knowledge, LLM schedulers [ 54 , 38 , 17 , 2,34 , 3, 42 , 59 , 53 , 11 ] follow the same methodology as tradi-tional request routing: upon receiving a request, the scheduler computes a score for each instance i and routes the request to the instance with the lowest (or highest) score, depending on the score definition. Figure 10 illustrates the methodology. Specifically, for each instance to route, its score is defined as 

f (ci, l i) over two per-instance indicators: ci estimates how much of the request’s prefill can be skipped by reusing in-stance i’s locally cached KV$ (e.g., the prefix hit length), and li estimates the instance’s load (e.g., its batch size or the number of inflight tokens). Our analysis uses the following representative methods: • BAILIAN is the production method currently deployed at B AILIAN . It uses a linear-combined scoring function of KV$-awareness (the hit ratio) and the current load (batch size). Other popular open-source schedulers adopt a similar design [ 34 , 3 ]. Specifically, the f is defined as 

f = (1 −λ)·ˆli +λ·(1 − ˆci) (the lower the better), where ˆci

is the KV$ hit ratio and ˆli is the batch size of the instance, and λ is a tuned weight that trades KV$ reuse for load balance. Note that each indicator carries a ˆ· that normalizes it to a common range; otherwise the two indicators cannot be added together. For serving agentic workloads, the operator explicitly sets a large λ to favor more KV$ hits. • LMetric [ 54 ] is the state-of-the-art scheduler published recently, whose effectiveness has been verified on non-agentic workloads. It uses a multiplication-based combinator to combine the KV$-awareness and load balance indicators: f = ( L − ci) × li, where L is the request’s prompt length, so L−ci is the prefill computation left after reuse. The multiplication eliminates the tuned weight while preserving both KV$-awareness and load balance. • Load-balance-only considers load only, i.e., f = li,routing each request to the least-loaded instance regardless of its KV$ reuse. It is a variant of the classic join-the-shortest-queue policy used in traditional cluster-level request routing and the default policy of vLLM [ 44 ], a popular LLM serving framework. 

The analysis methodology. We compare the performance of scheduling methods by replaying our production trace (Trace 1) on a 32-instance vLLM [ 44 ] cluster serving Qwen3-Coder-30B-A3B (detailed configurations in §5.1). Since the evaluated model generates responses that differ from those recorded in the trace, a strawman replay would distort the request patterns. We therefore carefully adjust the responses during replay to ensure a consistent trace execution, e.g., forcing the model to generate the same number of output tokens as recorded in the trace (detailed in §5.1). The cluster uses Mooncake [ 38 ] with LMCache [ 29 ], the best-performing open-source stack to our knowledge, as its global KV$ store. 6We carefully tuned each system to its peak performance: for example, under microbenchmarks, KV$ fetches from the global tier saturate the cluster’s RDMA bandwidth. We only replay Trace 1: the two traces exhibit similar patterns, as we have analyzed in §3, so we expect similar serving performance on Trace 2. We only report the results of one model served under PD colocation here; the results of PD disaggregation are similar and are presented in §5. We report each method’s maximum TPS within SLO . For each offered request rate, TPS within SLO is the cluster-wide TPS contributed only by requests that satisfy both the TTFT and TPOT SLOs. As discussed in the introduction, the request-level SLO remains important for agents: although an agent tolerates fluctuations in per-token latency, a response that is too slow makes the agent treat the request as timed out [ 7 ]. We set the SLO following B AILIAN ’s production targets, which also match the common practice in academic studies [ 58 , 46 , 55 ]: a TTFT SLO linear in the number of input tokens and a fixed TPOT SLO (30 ms). We sweep the offered request rate and take the maximum TPS within SLO over the sweep. Finally, we also analyze the performance with different global tier provisioning. We call the global tier fully provi-sioned when it is configured with the maximum capacity of our testbed. For the model analyzed in this section, the fully provisioned global tier is never the bottleneck: it holds the KV$ of all ongoing sessions, and its bandwidth keeps up with the fetch demand. Nevertheless, in production we do observe under-provisioned global tiers, due to either insufficient ca-pacity or bandwidth. We therefore additionally sweep the global tier’s per-node capacity, relative to the local tier, and report each method’s performance under each provisioning. 

Load balance does not always trade KV$ reuses. Counter-intuitively, the load-balance-only method—completely KV$-agnostic at scheduling time—recovers most of the lost TPS once the global tier is fully provisioned, as shown in Fig-ure 11 (a): it is within 7 % of the strongest baseline. Fig-ure 11 (b) explains why: with a fully provisioned global tier (red bars), all methods reach a nearly identical reuse ratio (73–75 %), because the KV$ missed in the local tier is fetched from the global tier. The methods differ only in where the reuse comes from: load-balance-only fetches most of its reused KV$ from the global tier (hatched), while B AILIAN 

hits mostly in the local tier (solid). Retrofit for the tail with pre-filter routing (Line 17 in Fig-ure 13). The not_overloaded() guard checks whether the target instance is overloaded. Retrofit for evicted sessions (Line 20 in Figure 13). The benefit of sticking requests from the same session to the same instance is to enjoy local-tier KV$ hits, but the local tier can 8evict the KV$, e.g., when a session idles long enough before its next turn arrives. Though rare, such a request should be treated as the first request of a fresh session, so SM ETRIC 

balances it across instances. SM ETRIC detects such a case using the actual hit length already computed for routing. Because each request car-ries its full conversation history through the LLM API [ 35 ], an unevicted follow-up request should produce a local hit that nearly covers that history. SM ETRIC therefore estimates the hit it expects from the request’s carried history— est_-hit , which excludes the freshly appended turn that is never cached—and sticks only when the best actual hit reaches at least HIT_RATIO of this estimate. HIT_RATIO is a hyper-parameter below one that absorbs the small drifts between the estimate and the real hit. If the best actual hit falls be-low HIT_RATIO × est_hit , the session’s KV$ has likely been evicted, and SM ETRIC falls back to load balancing. Our ablation (§5.5) shows SM ETRIC ’s performance is robust to this hyperparameter. 

The rationale of using the session turn and its availability. 

SM ETRIC relies on knowing each request’s turn in its session for scheduling (line 7). We choose the turn as the indicator because it can be collected in a router-stateless way—by stateless, we mean the router keeps no record of users’ past requests. A naive alternative is to have the router remember where each session executes—recording, for every active session, the instance it was routed to. This forces the router to maintain a session-to-instance table: the table grows with the number of active sessions, must be kept up to date, and must be garbage-collected as sessions fall idle, since a session never signals its end. Choosing the turn avoids all of this because it can be piggybacked or inferred from the LLM requests, as we describe below. We obtain the turn in this stateless way as follows. One simple option is to modify the agent framework to tag each request with its turn, but this raises compatibility issues: agent frameworks are diverse and evolve quickly (e.g., Claude Code [ 6] and OpenClaw [ 36 ]), and a provider can modify only its own in-house framework, not the external ones its users run. Instead, we infer the turn information: agents interact with LLM serving through standard APIs (e.g., the OpenAI API [ 35 ]), and since the API follows a stateless de-sign, each request carries the past conversations of the session, so SM ETRIC deduces the turn directly from the number of historical messages in the request. The deduction fails only if the agent drops the history, but the failure is benign: dropping the history forfeits its KV$ reuse, which is uneconomical for the agent and thus rare; even when it happens, SM ET - 

> RIC

simply treats the request as the first request of a new session—which is the right decision, since such a request hits little KV$ anyway. 

# 5 Performance evaluation 

5.1 Evaluation setup Testbed. Our testbed is a cluster of four GPU servers, each equipped with eight NVIDIA H20 GPUs (96 GB HBM3 per GPU), 160 CPU cores, and 1280 GB of host DRAM. The cluster thus supports up to 32 instances for single-GPU models. Servers are interconnected with 200 Gbps RDMA NICs. Each instance runs the latest vLLM [ 22 , 44 ] with de-fault optimizations enabled and tuned parameters. The global KV$ store is based on Mooncake [ 38 ] and LMCache [ 29 ], which implement the state-of-the-art global-tier architecture described in Figure 2. The scheduler runs on a separate CPU server with 64 cores and 256 GB DRAM. Unless otherwise stated, we use all instances to evaluate the performance. 

Evaluated model. We evaluate two models, Qwen3-Coder-30B-A3B and Qwen3-235B-A22B-FP8, both popular models on the market [ 48 ]. Qwen3-30B can be served with only one GPU, allowing us to evaluate up to 32 instances. In contrast, Qwen3-235B requires a minimum of 4 GPUs to serve as an instance, so we can evaluate up to 8 instances. 

Evaluated workload. We replay Trace 1 (§3) for all experi-ments. Because the requests are collected from a much larger cluster than our testbed, we sample them to control the request rate to the cluster, following prior work [ 54 , 58 , 31 , 19 , 41 ]. One issue with replaying is that the evaluated model’s re-sponses differ from those recorded in the trace. Such a differ-ence breaks the KV$ reuse pattern in two ways: the generated response may have a different length, and a follow-up re-quest’s recorded history no longer matches the tokens the serving instance actually cached. We fix the issue as follows. First, we force the evaluated model to generate exactly as many tokens as the recorded response, by ignoring the end-of-sequence token and capping the generation at the recorded length. Second, we rewrite a part of the follow-up requests with the generated content to ensure consistent KV$ reuse.

# 6 Related Work 

LLM request scheduling. Recent work on LLM serving has explored request scheduling from multiple perspectives, including KV$-aware and SLO-aware scheduling, dynamic rescheduling with request migration, and flexible request partitioning across instances [ 42 , 38 , 43 , 59 , 11 , 40 ]. Our work builds upon these insights and mechanisms, and fur-ther derives agent-session-aware hints for the scheduler to improve serving performance for agents. Unlike systems that co-design serving with agent applications and require appli-cations to expose inter-request structure [ 26 , 30 ], SM ETRIC 

operates entirely on the provider side, treating agents as un-modified clients. This matches how LLM providers, including BAILIAN , serve agents today. 

Serving KV$ efficiently. Reusing KV$ across requests 13 has been widely studied to avoid recomputing KV tensors for LLM serving, and both industry and academia have built mature system mechanisms for supporting KV$ reuse [ 15 ,16 , 22 , 49 , 50 , 57 , 52 ]. We build upon the mechanism of KV$ reuse, including multi-level KV$ store, while providing a systematic analysis and optimizations of the scheduling aspects for better utilization of the multi-level KV$ store. 

Other KV$-related optimizations. Besides cross-request reuse, recent work also optimizes the memory footprint, trans-fer cost, and placement of KV$ with a system–algorithms co-design. For example, CacheGen compresses and streams KV states to reduce KV$ transfer latency and bandwidth consumption [ 27 , 23 , 18 ]. KV quantization and compres-sion methods reduce KV$ memory pressure through low-bit quantization, token pruning, layer-wise allocation or unified compression [ 28 , 20 , 24 , 10 , 56 ]. Despite their potential per-formance advantages for KV$ reuse, these techniques com-promise model accuracy, which to the best of our knowledge has limited their adoption in production serving. Neverthe-less, these optimizations can strengthen several aspects of our session-centric balanced design if applied. For example, with a largely compressed KV$ size, GPU cache hits become more important, and SM ETRIC can achieve them in a load-balanced way. 

Optimizing LLM serving. Beyond cluster-level scheduling and KV$ management, recent work has optimized LLM serv-ing at different layers of the stack. For example, DeepSeek-V3 and DeepSeek-V4 reduce long-context inference cost through algorithm-architecture co-designs such as MLA and compressed sparse attention [ 12 , 13 ], FlashInfer [ 51 ] im-proves attention execution through customizable kernels and optimized KV layouts, BlitzScale and Aegaeon improve elas-ticity and resource scaling for serving workloads [ 48 , 55 ], and many others [ 21 , 51 , 25 , 22 ]. These optimizations are orthogonal to SM ETRIC .

# 7 Conclusion 

Agentic serving shifts the goal of LLM scheduling to cluster-wide TPS and makes KV$ reuse dominant. Our study of two real-world traces shows that existing schedulers trade load balance for KV$ reuse, and that this trade is unnecessary: the global-tier KV$ store decouples reuse from request place-ment, and the workload’s intra-session locality means balanc-ing each session’s first request suffices to balance the cluster. SM ETRIC turns these insights into balanced session-centric scheduling—first requests scheduled for balance, follow-up requests routed cache-aware—and outperforms state-of-the-art schedulers: 10–16 % higher TPS under prefill-decode colocation with a global store and 2–34 % higher prefill TPS under disaggregation across global-tier provisionings.
