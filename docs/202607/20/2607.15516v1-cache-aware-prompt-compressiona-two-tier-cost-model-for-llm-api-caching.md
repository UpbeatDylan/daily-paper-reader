## 摘要
生产级LLM部署结合了两种成本降低策略：提示缓存（对重复使用的令牌前缀提供折扣费率）和提示压缩（发送更少的令牌）。压缩文献已标准化为查询感知方法，该方法为每个查询生成不同的压缩前缀，从而在每次调用时机械地使前缀严格缓存失效。我们在Anthropic的Sonnet 4.6 API上实证评估了这种成本，发现缓存远非文献假设的rho=1.0理想状态：Sonnet的缓存具有两层架构，在约3500个令牌处存在一个尖锐阈值，低于该阈值时，在30次调用会话中命中率稳定在rho~0.83。我们的成本模型预测，并在实验中确认，在现实rho下，查询感知压缩在高压缩比（r>=6）下优于朴素缓存。我们提出缓存感知的提示压缩（CAPC），将查询无关压缩与显式cache_control以及一个层级保持比率界限相结合，防止过度压缩将缓存前缀推入热层级。CAPC是LongBench-v2上16/16配置中最经济的策略，相比纯缓存平均节省49%，相比查询感知压缩节省64%，相比普通版本节省90%，质量与未压缩基线相差在0.05以内。我们在三个生产工作负载上验证了CAPC：一个企业工具使用助手，带有94k令牌模式前缀（r=3时成本降低51.7%）；一个跨两个代码库的图化知识图谱RAG管道（在FastAPI上比全缓存快9.3倍，在httpx上快2.4倍）；以及公共tau-bench零售基准（50个任务），其中CAPC是四种策略中最经济的，奖励恰好等于普通版本（均为36/50，p=1.00），而查询感知压缩是最昂贵的，比普通版本高40.1%——这是首次在公共基准上验证交叉模型的负ROI预测的生产环境证据。

## Abstract
Production LLM deployments combine two cost-reduction primitives: prompt caching (a discounted rate for re-used token prefixes) and prompt compression (fewer tokens sent). The compression literature has standardized on query-aware methods that produce a different compressed prefix per query, mechanically invalidating the prefix-strict cache on every call. We characterize this cost empirically on Anthropic's Sonnet 4.6 API and find caching is far from the rho=1.0 ideal the literature assumes: Sonnet's cache has a two-tier architecture with a sharp threshold near 3,500 tokens, below which the hit rate plateaus at rho~0.83 across 30-call sessions. Our cost model predicts, and experiments confirm, that under realistic rho, query-aware compression beats naive caching at high compression ratios (r>=6). We propose Cache-Aware Prompt Compression (CAPC), pairing query-agnostic compression with explicit cache_control plus a tier-preserving ratio bound that prevents over-compression from pushing the cached prefix into the hot tier. CAPC is the cheapest strategy in 16/16 configurations on LongBench-v2, with mean savings of 49% over cache-only, 64% over query-aware compression, and 90% over vanilla, at quality within 0.05 of the uncompressed baseline. We validate CAPC on three production workloads: an enterprise tool-using assistant with a 94k-token schema prefix (51.7% cost reduction at r=3); a graphify knowledge-graph RAG pipeline across two codebases (9.3x vs cache-all on FastAPI, 2.4x on httpx); and the public tau-bench retail benchmark (50 tasks), where CAPC is the cheapest of four strategies with reward exactly equal to vanilla (both 36/50, p=1.00) while query-aware compression is the most expensive at +40.1% over vanilla -- the first production confirmation of the crossover model's negative-ROI prediction on a public benchmark.

# 论文总结：Cache-Aware Prompt Compression: A Two-Tier Cost Model for LLM API Caching

## 1. 论文的核心问题与整体含义（研究动机和背景）

*   **核心冲突**：大型语言模型（LLM）的生产部署常使用两种成本削减策略：**提示缓存**（Prompt Caching，对重复使用的令牌前缀提供折扣）和**提示压缩**（Prompt Compression，减少发送令牌数量）。然而，这两者存在根本性矛盾。
*   **现有范式缺陷**：提示压缩领域的标准方法——**查询感知压缩**（Query-aware Compression）——每次会为不同的查询生成不同的压缩前缀。这导致每次调用都使针对前缀的缓存机制失效，每次调用都变成缓存未命中（cache miss），从而支付全额的非缓存输入价格。
*   **研究动机**：该论文旨在量化和解决这一冲突。文献中通常隐含假设缓存命中率 ρ = 1.0（即总能命中），但论文通过对 Anthropic Sonnet 4.6 API 的实际测量，揭示了现实世界中的缓存其实表现远非如此理想。

## 2. 论文提出的方法论：核心思想、关键技术细节

论文的核心方法论包含三个部分：

*   **经验性缓存架构描述**：论文通过实验，实证刻画了 Anthropic Sonnet 4.6 的提示缓存行为：
    *   **两级缓存架构**：存在一个约 3500 个令牌（token）的“热/持久”两级阈值。前缀大小低于此阈值时，命中率 ρ 显著下降（约 0.83）；高于阈值时，命中率接近 1.0。

*   **现实成本模型**：论文提出了一个更现实的成本模型 `cost(N, |P|, r)`，该模型包含了真实的、与令牌数相关的缓存命中率。
    *   **策略定义**：定义了四种策略：
        *   A (Vanilla): 无缓存、无压缩。
        *   B (Cache-only): 完整文档使用缓存，无压缩。
        *   C (Query-aware compress): 每次查询使用不同的压缩前缀，无缓存。
        *   D (CAPC): 查询无关压缩 + 显式缓存。
    *   **交叉分析**：推导出一个通用交叉阈值公式 `ρ_cross(r) = (1 - 1/r) / (1 - β/α)`，用来预测在何种压缩比下，缓存策略会输给压缩策略。在该实例中，当压缩比 r >= 6 时，查询感知压缩的收益会超越朴素缓存。

*   **核心算法：CAPC**
    *   **核心思路**：采用**查询无关压缩**（Query-agnostic Compression），对静态文档进行一次性压缩。由于所有查询使用相同的压缩前缀，该前缀可以被缓存复用，从而避免了缓存失效。
    *   **关键机制**：
        1.  **缓存感知的提示压缩 (CAPC)**：先将文档 `D` 压缩成 `D'`（离线处理一次），然后将 `D'` 作为缓存前缀写入，并添加 `cache_control` 指令。
        2.  **层级保持比率界限 (Tier-preserving Ratio Bound)**：设计了一个安全压缩比 `r_safe = floor(|D| / 3500)`，防止因过度压缩，导致压缩后前缀长度低于 3500 令牌阈值而掉入“热层级”，造成命中率大幅下降和成本反弹。
        3.  **自适应缓存边界算法**：对于会演变的文档，通过观察不同版本的变异率，自动识别文档中的“静态”（STATIC）、“准静态”（QUASI）和“动态”（DYNAMIC）部分，仅缓存连续稳定的前缀部分，以最大化缓存收益。

## 6. 论文的主要结论与发现

1.  **缓存真实表现非理想**：现实世界的 LLM API 缓存具有两层架构，且命中率取决于前缀大小和调用次数，文献中的 `ρ=1.0` 假设不成立，尤其是在小前缀上。
2.  **交叉效应**：在较高压缩比下（r>=6），查询感知压缩策略的成本会低于纯缓存策略，这与传统认知相反。
3.  **CAPC 的成本优势**：论文提出的 CAPC 在所有实验中都表现出显著的优越性：
    *   在 LongBench-v2 上，是所有配置中最经济的（平均比缓存节省 49%，比查询感知压缩节省 64%，比普通版本节省 90%）。
    *   在生产 EA 助手场景下，成本降低 **51.7%**。
    *   在知识图谱 RAG 中，成本降低 **9.3x** (FastAPI)。
    *   在 τ-bench 中，是成本最低的策略，同时奖励与普通版本一致。
4.  **质量保持**：在大多数情况下，CAPC 可以在显著降低成本的同时，保持与无缓存普通版本几乎一致的质量。
5.  **查询感知压缩的负ROI**：论文首次在公共基准（τ-bench）上证实了查询感知压缩可能带来负ROI，其成本比普通版本高 40.1%，因为它的缓存失效效应抵消了压缩带来的令牌节省。
6.  **先验知识的关键作用**：在 RAG 应用中，CAPC 的增益与模型对知识库的“先验知识”成反比。模型知道得越少，CAPC 带来的质量提升越大。
