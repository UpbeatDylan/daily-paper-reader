## 摘要
随着大语言模型（LLM）推理从纯文本范式向多模态范式演进，推理系统面临三大挑战：（1）多模态工作流的灵活编排，其中异构计算单元呈现复杂依赖关系和并发控制；（2）跨进程与节点的高效海量中间数据传输，张量在异构角色间高速流动；（3）跨角色的KV缓存与模型权重高效共享，以消除冗余GPU内存。现有解决方案独立部署LLM和扩散模型，缺乏多模态管线的系统级抽象，导致编排逻辑分散、传输路径与特定模型紧密耦合，且集成新模型成本高昂。针对这些挑战，我们提出Omni-Flow，一种通过三层抽象实现多模态推理的分布式调度框架。控制流层通过Python DSL定义工作流，将异构单元编排为统一数据流图，支持静态DAG和动态路由，并内置服务发现及多种负载均衡策略。数据流层提供超越预填充/解码分离的分布式KV缓存抽象，统一分配并通过零拷贝低延迟通道实现跨三层分页存储层级（GPU/CPU/SSD）的直接跨角色传输。计算流层支持多轮对话中KV复用的复杂多模态前缀匹配，并通过统一SGLang接口接管KV缓存与采样逻辑，使扩散模型在统一并行语义下直接复用LLM前向路径。我们证明Omni-Flow以一致的编程模型支持多种异构场景，包括全模态对话（LongCat-Next）和复杂图像生成管线（HunyuanImage-3）。

## Abstract
As large language model (LLM) inference evolves from text-only to multimodal paradigms, inference systems face three challenges: (1) flexible orchestration of multimodal workflows, where heterogeneous computing units exhibit complex dependencies and concurrent control; (2) efficient transmission of massive intermediate data across processes and nodes, with tensors flowing at high speed among heterogeneous roles; and (3) efficient sharing of KV caches and model weights across roles to eliminate redundant GPU memory. Existing solutions deploy LLMs and diffusion models independently, lacking a system-level abstraction for multimodal pipelines; this scatters orchestration logic, tightly couples transmission paths to specific models, and incurs high cost to integrate new models. To address these challenges, we present Omni-Flow, a distributed scheduling framework for multimodal inference through a three-layer abstraction. The Control Flow layer defines workflows via a Python DSL, orchestrating heterogeneous units into a unified dataflow graph that supports static DAGs and dynamic routing, with built-in service discovery and diverse load-balancing strategies. The Data Flow layer provides a distributed KV cache abstraction beyond prefill/decode separation, unifying allocation and enabling direct cross-role transmission across a three-tier paged storage hierarchy (GPU/CPU/SSD) over zero-copy, low-latency channels. The Compute Flow layer supports complex multimodal prefix matching for KV reuse across multi-turn dialogues, and takes over KV cache and sampling logic via a unified SGLang interface, letting diffusion models directly reuse the LLM forward path under unified parallel semantics. We demonstrate that Omni-Flow supports diverse heterogeneous scenarios with a consistent programming model, including omni-modal dialogue (LongCat-Next) and complex image generation pipelines (HunyuanImage-3).

## 论文提出的方法论：核心思想与关键技术细节

### 3. 整体架构：三层抽象（Control Flow、Data Flow、Compute Flow）
- **Control Flow（控制流）**：提供Python DSL声明式定义工作流，将多模态推理建模为有向图（支持循环），节点代表异构计算角色，边代表数据传输通道。支持OR-AND输入语义、流式帧级激活、运行时动态分支选择（基于输出类型自动路由）、多帧分段生产等高级语义。
- **Data Flow（数据流）**：统一管理中间张量和分布式KV缓存，构建**全局参数池（Global Params Pool）**，实现L1 GPU / L2 CPU / L3 SSD三级分页存储，支持自动冷热迁移。采用Client-Server架构，MemoryManager统一管理内存池，通过ZMQ IPC和RDMA实现零拷贝跨节点传输。另有**Tmp Buffer Pool**管理临时张量，**Weight Share**实现跨角色权重共享。
- **Compute Flow（计算流）**：基于SGLang深度定制，支持复杂多模态前缀匹配（完全匹配和增量匹配），通过统一接口接管KV缓存池和采样逻辑。对于兼容的扩散模型（DiT），直接复用LLM的PagedAttention基础设施和前向执行路径，共享权重和并行语义。

### 关键技术细节

#### 控制流层：三种依赖类型的处理机制
- **钻石依赖（Diamond Dependency）**：多路径数据汇聚时，框架通过哈希计算激活ID（activation id），确保同一源的多路径落入同一桶（Bucket），不同帧落入不同桶。桶维护`required`、`received`、`slot_eos`、`unreachable`集合，当所有必要路径都就绪或不可达时触发前向计算。
- **序列依赖（Sequence Dependency）**：由SessionLoadBalance节点维护期望序列号，仅在序列号匹配时允许通过，且等待当前OR组完结后推进序列号。非SLB节点完全无序执行，避免引入冗余串行化点。
- **多流连接（Multi-Stream Join）**：通过`activate by count`声明，框架将多路输入强制归入同一桶，按字段级帧数阈值触发前向计算。

#### 全局参数池（KV缓存管理）
- **物理分页层**：支持灵活的元数据结构（sub_name维度和slice_name维度），适应不同Transformer层和注意力头配置。
- **多级槽引擎**：基于TinyLFU + 四级频率感知LRU驱逐策略；使用自定义CUDA内核实现GPU NHD布局与CPU/SSD序列化布局之间的零拷贝格式转换；并发缺失请求通过飞行中上传字典去重。
- **分布式元数据协调**：基于Redis的Lua脚本保证原子性，包含布局ID标识、会话前缀匹配、副本发布与选择、三级保护原子驱逐（副本引用锁、会话运行锁、候选组合完整性）。
- **完整数据流闭环**：初始化→推理写入→发布通知→预加载→跨节点RDMA拉取→写时复制与碎片页处理→内存压力下驱逐。

#### 前缀匹配与KV复用
- **完全匹配**：运行所有模态编码器，拼接完整序列，逐页前缀匹配；编码器维护LRU缓存避免重复计算。
- **增量匹配**：客户端携带步进偏移，服务器截断历史仅处理增量数据。
- 两种模式共享同一链式分页哈希机制，将基数树语义压缩为可跨进程比较的哈希字符串。
- LLM前缀复用分解为两层：**层1**（可重算嵌入，存储于CPU）作为安全网；**层2**（KV缓存命中，尽力而为）仅当GPU缓存命中时受益。

#### 扩散模型复用LLM路径
- DiT不采用原生扩散运行时，而是继承LLM的KV-权重管理基类，通过环境切换选择具体实现。DiT去噪步骤组织为序列，沿标准PagedAttention路径执行，图像和时间步数据作为额外字段附加到前向批次中。此举对齐了TP/EP并行分割语义，使权重和KV缓存能够直接共享。

### 算法/流程（文字说明）
- **Bucket模型**：框架在收敛节点分配桶，桶由四元组（gc_rid, owner_node, or_group, activation_id）唯一标识。activation_id通过基于共同分叉源身份和帧序列号的哈希计算，确保同源同帧数据落入同一桶。桶的触发条件为`required ⊆ received ∪ unreachable`。桶懒加载、即时回收。
- **前缀匹配**：完全匹配模式下，服务器运行所有编码器得到精确段长度，拼接隐藏状态后逐页匹配；增量匹配模式下，根据客户端步进偏移截断历史，仅处理新增量。两种模式共用链式分页哈希。
- **驱逐与保护**：驱逐检查通过单个Lua脚本原子执行，确保副本引用锁、会话运行锁、候选组合完整性全部通过后才实际删除。
