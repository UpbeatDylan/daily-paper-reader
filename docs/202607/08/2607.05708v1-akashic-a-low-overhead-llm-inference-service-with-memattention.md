## 摘要
近期基于LLM的智能体系统在多轮交互、工具调用和跨会话工作流中不断累积上下文。为每个请求重放完整历史很快变得不切实际：长上下文增加预填充成本，可能超出上下文限制，并且常常将任务相关证据淹没在不相关内容中，降低服务效率和输出质量。我们提出Akashic，一个围绕MemAttention构建的低开销记忆系统，它将上下文组织成有界块并在块间建模语义关系，无需反复重写完整历史即可保留跨块证据。Akashic进一步应用硬件-软件协同设计的内存放置，将可能共同检索的块共置，减少检索碎片和I/O开销。在四个代表性工作负载和三种模型规模上，与强大的先验记忆基线相比，Akashic将任务准确率提升高达10.2个百分点，吞吐量提升高达1.21倍，可持续请求率提升高达1.88倍。

## Abstract
Recent LLM-based agent systems continuously accumulate context across multi-turn interactions, tool invocations, and cross-session workflows. Replaying the full history for every request quickly becomes impractical: long contexts increase prefill cost, may exceed context limits, and often bury task-relevant evidence in irrelevant content, degrading both serving efficiency and output quality. We propose Akashic, a low-overhead memory system built around MemAttention, which organizes context into bounded chunks and models semantic relationships across chunks, preserving cross-chunk evidence without repeatedly rewriting the full history. Akashic further applies hardware-software co-designed memory placement to co-locate likely co-retrieved chunks, reducing retrieval fragmentation and I/O overhead. Across four representative workloads and three model sizes, Akashic improves task accuracy by up to 10.2 points, throughput by up to 1.21x, and sustainable request rate by up to 1.88x over strong prior memory baselines.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：LLM agent系统在多轮交互、工具调用和跨会话工作流中不断累积上下文。为每个请求重放完整历史导致预填充（prefill）成本高、易超出上下文窗口限制，且无关内容会干扰模型输出，降低服务效率和任务准确性。
- **背景**：现有记忆系统（如Mem0、MemGAS、MemGPT、RMM）多采用全局总结或分段检索，面临**粒度权衡**——粗粒度总结混合无关主题、细粒度分段丢失跨段依赖；同时存在**局部性差距**（locality gap）：语义相关的记忆物理上可能分散在不同页面，导致检索I/O开销大。
- **整体含义**：需要一种既能保持高质量记忆（准确恢复跨块证据）又能高效服务（低维护开销、低检索延迟）的记忆系统。

## 2. 方法论

### 核心思想
Akashic提出**MemAttention**，将上下文组织为**有界块**，采用**增量块级维护**而非全局压缩，通过**跨块推理**（cross-chunk inference）保留跨块语义关系，并利用**硬件-软件协同设计**的内存管理器优化物理局部性。

### 关键技术细节
- **块级内存构建**：输入上下文被切分为固定大小（默认1024 token）的块。仅当活动块达到阈值（`chunk_gate_tokens`）时，才对该块进行一次压缩，维护复杂度为 `O(L_chunk)` 而非 `O(总历史长度)`。
- **跨块协调**：压缩新块时，将其元数据与已压缩块的元数据一同送入LLM进行语义匹配，选出最相关的top-5旧块进行联合压缩（joint compaction），从而更新或淘汰过时记忆。
- **统一语义匹配机制**：在维护阶段和推理检索阶段均使用同一模型驱动机制（LLMSelect），基于元数据关键词选择相关块。推理时选出top-5相关块并拼接到当前提示中。
- **内存管理器**（Memory Manager）：
  - **在线重定位**：在联合推理后，利用模型推断块间共访问亲和性，将可能共同访问的块写到新块中，原拷贝标记无效（就地更新）。
  - **后台垃圾回收**：当块无效比 `φ(B) >= 0.75` 时，提取存活块，按用户/会话分组，再按模型推断的亲和性重新打包为新块，释放旧块。从而减少未来跨块推理的物理读次数。

### 算法流程（文字说明）
- **MemAttention算法**：
  1. 将新轮次追加到活动块`C_a`。
  2. 若`|C_a| >= τ_c`，则压缩该块并提取元数据关键词。
  3. 收集同一namespace下的先前块元数据，由LLM选出top-5相关块进行联合压缩，更新/删除过时记忆。
  4. 将结果写回块表，重置活动块。
  5. 推理时，从当前上下文提取查询关键词，再次由LLM从块表元数据中选出top-5相关块，按时间排序后拼接到当前提示前。
- **内存管理器算法**：
  - 联合推理路径：对检索出的块集，用LLM识别应共置的子集，离地重写到新块，更新目录并标记旧拷贝无效。
  - 后台压缩路径：扫描无效比≥0.75的块，提取存活块，按用户/会话分区，每个分区内用LLM分组亲和块，再打包为新块并释放原块。

- Akashic在所有九种设置下均处于Pareto前沿，同时实现高任务质量和高服务效率。

## 7. 优点

- **创新性**：将长程记忆构建从全局压缩转变为增量块级维护，解决了粒度权衡问题。
- **统一性**：维护和检索使用同一套LLM驱动的语义匹配机制，无需不同相似度函数。
- **系统工程视角**：明确识别并解决了“局部性差距”，设计了硬件-软件协同的内存放置方案，结合在线重定位和后台GC，兼顾局部性优化和存储开销。
- **全面实验**：覆盖多种工作负载、模型规模和并发压力，消融和敏感性分析完整，证明设计的有效性和鲁棒性。
- **低侵入实现**：仅修改vLLM控制路径，不改变GPU执行路径，便于实际部署。

## 8. 不足与局限
