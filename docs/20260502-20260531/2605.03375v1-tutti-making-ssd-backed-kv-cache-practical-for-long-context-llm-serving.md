## 摘要
LLM服务依赖前缀缓存来提升推理性能。随着上下文不断增长，键值（KV）缓存占用远超GPU HBM和CPU DRAM容量，KV缓存逐渐被卸载到NVMe SSD。然而，从SSD恢复KV缓存存在I/O性能差的问题，并导致严重的GPU停顿。这主要是因为碎片化的GPU内存布局导致大量细粒度的随机I/O，即使采用GPU Direct Storage（GDS），低并行度的CPU仍然是严重瓶颈，因为GDS仍依赖CPU介入来发起每个I/O，因此本质上是以CPU为中心的。本文提出Tutti，一种高效的SSD支持的KV缓存解决方案，消除了CPU在HBM和SSD之间的关键数据和I/O控制路径中的干预。Tutti的核心是一个以GPU为中心的KV缓存对象存储，其中CPU仅负责每层一次将I/O内核异步加载到GPU。Tutti通过以下设计饱和NVMe SSD带宽并将GPU停顿降至接近零：（i）提供GPU原生的对象抽象，支持批量KV缓存传输和管理；（ii）通过引入GPU io_uring重新架构GPU存储栈，支持异步GPU直接对象I/O；（iii）提出松弛感知的I/O调度以避免GPU资源争用。我们实现了Tutti并将其集成到vLLM。大量评估表明，与最先进的启用GDS的SSD支持的LMCache相比，Tutti在严格的SLO约束下将TTFT降低了78.3%，并将可达到的请求率提高了2倍。服务成本降低了27%。Tutti实现了与DRAM支持的LMCache几乎相同的推理性能，同时提供了几乎无限的容量。

## Abstract
LLM serving relies on prefix caching to improve inference performance. As growing contexts push key-value (KV) cache footprint far beyond GPU HBM and CPU DRAM capacity, KV cache is increasingly offloaded to NVMe SSDs. Unfortunately, restoring KV cache from SSDs suffers from poor I/O performance and incurs significant GPU stalls. This is primarily because the fragmented GPU memory layout results in a massive number of tiny random I/Os, rendering the low-parallelism CPU a severe bottleneck even with GPU Direct Storage (GDS), which still relies on CPU intervention to initiate each I/O and thus remains CPU-centric. This paper presents Tutti, an efficient SSD-backed KV caching solution that eliminates CPU intervention from the critical data and I/O control paths between HBM and SSDs. At the core of Tutti is a GPU-centric KV cache object store, in which the CPU is only responsible for asynchronously loading I/O kernels once per layer to the GPU. Tutti saturates NVMe SSD bandwidth and reduces GPU stalls to near zero through the following designs: (i) we provide a GPU-native object abstraction that enables bulk KV cache transfers and management; (ii) we re-architect the GPU storage stack by introducing GPU io_uring to support asynchronous GPU direct object I/O; and (iii) we propose slack-aware I/O scheduling to avoid GPU resource contention. We have implemented Tutti and integrated it to vLLM. Extensive evaluation shows that compared to the state-of-the-art GDS-enabled, SSD-backed LMCache, Tutti reduces TTFT by 78.3% under strict SLO constraints and improves the achievable request rate by 2x. The serving cost is reduced by 27%. Tutti achieves nearly the same inference performance as DRAM-backed LMCache, while providing almost infinite capacity.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：LLM 服务依赖前缀缓存（KV cache）来避免重复计算，但随着上下文窗口和并发量增长，KV 缓存远超 GPU HBM 和 CPU DRAM 容量，不得不卸载到 NVMe SSD。然而，从 SSD 恢复 KV 缓存时，由于 GPU 内存采用分页碎片化布局（如 vLLM 的 64-token 块），导致大量随机小 I/O，CPU 即使借助 GPU Direct Storage（GDS）也因每次 I/O 需 CPU 发起而成为瓶颈，造成 70%～80% 的 GPU 停滞，最终使得 KV 缓存重用比重新计算更慢。
- **现状**：现有三级 HBM-DRAM-SSD 缓存系统（如 LMCache）中，DRAM 层性能接近 HBM，但容量有限；SSD 层即使使用 GDS 仍因 CPU 介入导致 I/O 效率低下，GPU 气泡时间占比超 70%。
- **含义**：需要一种消除 CPU 关键路径、让 GPU 直接管理 SSD I/O 的方案，以在近乎无限容量的前提下达到接近 DRAM 的推理性能。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：构建以 GPU 为中心的 KV 缓存对象存储，CPU 仅负责每层一次异步加载 I/O 内核到 GPU，使 GPU 能直接向 SSD 发起大量并行 I/O 请求，从而消除 CPU 在数据和 I/O 控制关键路径上的干预。
- **关键技术细节**：
  - **GPU 原生对象抽象**：将每个 KV 块映射为一个对象，GPU 文件按层组织（每层一个 Key 对象和一个 Value 对象），通过可扩展的 GPU 文件池和基于 SGL（Scatter Gather List）的 P2P 内存映射表实现批量传输和管理。CPU 侧接口将分配、索引和高并发 GPU 访问合并为单次操作，将 CPU 开销从 O(层数×块数) 降至 O(层数)。
  - **GPU io_uring（gio_uring）**：模拟 CPU 侧 io_uring，在 GPU HBM 中维护锁自由的环缓冲区（SQ/CQ），CPU 预填充 I/O 控制块（IOCB，每个包含 2048 个 I/O 上下文）。通过 SM 分区（NVIDIA Green Context）将 GPU 资源划分为“计算域”和“I/O 控制域”，避免 I/O 内核阻塞计算内核。异步处理流程：`init_queue` → `get_iocb` → `issue_io` → `wait_cqe`。
  - **松弛感知 I/O 调度**：通过离线 profiling 记录每层在不同输入长度和前缀长度下的松弛窗口（空闲 SM 资源且无读写带宽冲突的时间段），运行时根据查找表决策何时启动读/写 I/O 内核。读写操作解耦，读优先，写延迟到后续松弛窗口或 decode 阶段，避免并发读写导致的带宽下降（实验显示并发读写总带宽下降 60%）。

- **Benchmark**：端到端延迟（TTFT 和 ITL）、原始 I/O 带宽、GPU 气泡时间、服务成本。

  - LMCache-SSD：通过 memcopy 和异步 I/O 卸载到 NVMe SSD。

## 6. 主要结论与发现

- 相比最先进的 LMCache-GDS，Tutti 在严格 SLO 约束下将 TTFT 降低 78.3%，可达请求率提升 2 倍，服务成本降低 27%。
- Tutti 实现了接近 DRAM-backed LMCache 的推理性能，同时提供几乎无限的 SSD 容量。
- 原始 I/O 带宽：Tutti 达到 25.9 GB/s（检索）和 ~10 GB/s（存储），远超 LMCache-GDS 的 11.9 GB/s 和 7 GB/s。
- 逐层异步流水线将“有效零气泡区”扩展至 98.3% 的缓存命中率，远超 SSD 基准的较低阈值。
- 在多 GPU 场景下，Tutti 能处理 640K 前缀长度（LMCache-GDS 因 OOM 失败），TTFT 仅 1.2 秒。
- SGL 相比 PRP 提升读写带宽 31 倍和 91 倍。

## 7. 优点

- **创新性**：首次提出 GPU 主导的 SSD-backed KV 缓存方案，彻底消除 CPU 关键路径，思路新颖。
- **技术整合**：巧妙组合 GPU 原生对象抽象、GPU io_uring 和松弛感知调度，解决了抽象不匹配、粒度差距和资源竞争三大挑战。
- **实验全面性**：覆盖多个维度（延迟、带宽、成本、可扩展性），并且对比了不同软件版本和缓存命中率，消融实验扎实。
- **实用性**：集成到 vLLM，兼容主流推理引擎，代码开源（~9500 行 C++/Python），部署成本低。
- **性能显著**：在严格 SLO 下大幅提升 QPS，降低服务成本，且能处理极端长上下文（640K tokens）。

## 8. 不足与局限

- **实验覆盖有限**：
  - 仅测试了 Llama3-8B 和 GLM-4-9B-1M 两个模型，缺乏更大模型（如 70B/405B）或 MoE 架构的验证。
  - 多 GPU 实验仅使用 2 GPU，未展示 8+ GPU 集群下的扩展性。
  - 未评估远程检索路径（当前仅支持本地 SSD，远程通过 CPU+RDMA 回退，性能可能退化）。
- **偏差风险**：
  - 离线 profiling 假设模型和硬件配置固定，动态负载变化（如突发请求）可能导致松弛估计失效。
  - 成本计算采用典型云定价，但实际价格因供应商和折扣而异。
- **应用限制**：
  - 依赖 NVIDIA Green Context 和 NVMe PRP/SGL 特性，对非 NVIDIA GPU 或非 NVMe 存储不兼容。
  - 需要预先分配大池的 GPU 文件，启动开销较高。
  - 当前不支持跨节点共享 SSD 池（需要 Mooncake 作为分布式协调层，但远程路径未优化）。
- **缺少消融实验**：未单独评估 GPU io_uring 本身相对于 CPU io_uring 的性能提升，松弛感知调度与简单流水线的对比不够深入。
