## 摘要
KV缓存复用机制日益暴露优先级、持续时间、卸载、路由提示、调度器模式和事件流。这些机制有助于保留可复用的前缀，但当常驻KV与活动实存KV无法同时容纳时，它们本身并未定义一种可移植的、针对已接受的未来复用状态的契约。我们引入常驻KV声明，这是一种将未来复用意图绑定到具体化谓词、生命周期状态、活跃/常驻可行性结果以及声明级遥测的合规契约。在受控的vLLM分配器探测中，一个60块的常驻声明和70块的活跃预填超过了80块的可用的KV池。写入拒绝（write no-admit）阻止了活跃请求成为未来的可复用状态，但仍允许活跃分配驱逐常驻块离开共享池。一个最小化的vLLM原型表明，硬保护常驻声明将此失败模式转化为调度器可见的活跃拒绝，并具有直接的阻塞声明归因。结果并非生产加速或新的缓存替换算法，而是一个运行时契约，将未报告的常驻丢失转化为可重构的活跃/常驻仲裁。配套的MicroRuntime和vLLM石蕊测试集区分了普通驱逐、软优先级、写入拒绝、已接受硬声明、具体化失败、降级、过期、活跃拒绝以及跟踪级结果重构。

## Abstract
KV-cache reuse mechanisms increasingly expose priority, duration, offload, routing hints, scheduler modes, and event streams. These mechanisms help preserve reusable prefixes, but they do not by themselves define a portable contract for accepted future-reuse state when resident KV and active live KV cannot both fit. We introduce resident KV claims, a conformance contract that binds future-reuse intent to a materialization predicate, lifecycle state, active/resident feasibility outcome, and claim-level telemetry. In controlled vLLM allocator probes, a 60-block resident claim and a 70-block active prefill exceed an 80-block usable KV pool. Write no-admit prevents the active request from becoming future reusable state, but it still allows active allocation to evict residents from the shared pool. A minimal vLLM prototype shows that hard protected resident claims convert this failure mode into scheduler-visible active refusal with direct blocking-claim attribution. The result is not a production speedup or a new cache-replacement algorithm. It is a runtime contract that turns unreported resident loss into reconstructable active/resident arbitration. A companion MicroRuntime and vLLM litmus suite distinguish ordinary eviction, soft priority, write no-admit, accepted hard claims, materialization failure, demotion, expiry, active refusal, and trace-level outcome reconstruction.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：在大语言模型服务中，KV缓存复用（如前缀缓存）有助于避免重复计算，但现有机制（优先级、TTL、卸载、调度模式等）只提供了“策略”（policy），而缺少一个**可移植的契约**来定义：当**常驻未来可复用KV**（resident reusable KV）与**当前请求的活KV**（active live KV）无法同时放入可用内存池时，运行时应该做什么？
- **动机**：现代运行时（如TensorRT-LLM、SGLang、vLLM）虽然暴露了大量保留原语，但这些原语在遇到内存不足时，通常默认通过普通缓存驱逐来牺牲常驻KV，而不会显式通知应用层。论文认为这是语义上的缺失——常驻KV不应该被当作普通缓存项，而应该是一种“承诺”（claim），一旦接受，必须通过明确的生命周期事件（拒绝、降级、过期、卸载、伤害）才能改变状态。
- **整体含义**：本文定义了**常驻KV声明（Resident KV Claims）** 这个合规契约，将未来复用意图绑定到具体化谓词、生命周期、可行性检查与声明级遥测。它不是一种新的缓存替换算法或生产加速方案，而是一个**运行时合同**，使系统能够将未报告的常驻KV丢失转化为可观察、可归因的活跃/常驻仲裁结果。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：
  1. 区分三种不同资源：常驻可复用KV、活跃实存KV、未来可复用准入（决定是否将新产生的KV加入可复用缓存）。
  2. 定义**常驻声明（ResidentClaim）**：一个应用可见的未来复用对象，提交给运行时，包含声明ID、拥有者范围、缓存等价标识、具体化谓词、内存占用估计、保护模式、可选持续时间。运行时接受后承担义务：保存该声明的具体化谓词，除非显式发出拒绝、降级、卸载、过期或伤害事件。
  3. 保护模式包括：soft_priority（软优先级）、hard_protected（硬保护，不能破坏）、demotable（可降级）、offloadable（可卸载）、expiring（到期终止）、best_effort（仅遥测）。

  5. 可行性边界：`protected_resident_kv + active_live_kv <= usable_kv`。若不满足，运行时必须选择显式动作（拒绝活跃请求、降级/过期常驻、卸载、路由等），并报告声明级遥测。

  - **写不接纳（Write No-Admit）**：一种控制行为——活跃请求被服务，但其产生的KV不被加入未来可复用缓存。论文证明这对常驻保护无效，因为活跃分配仍从同一物理池拉取。

## 6. 论文的主要结论与发现

1. **保留块量不等于有用价值**：Naive fair share保留了1104个缓存token，但阈值化价值为0；完整前缀策略保留更少token却获得高价值。有用常驻取决于具体化形状（前导连续前缀）。
2. **写不接纳不能保护常驻KV**：写不接纳只阻止未来复用，但不影响活跃分配；共享物理池下活跃分配仍会驱逐常驻块。
3. **可行性边界强制显式仲裁**：当60常驻+70活跃 > 80可用时，硬保护能使活跃分配失败并产生调度器可见的拒绝，而普通方法则静默丢失常驻KV。

6. **核心贡献是运行时合同而非性能提升**：论文明确不声称生产加速或p95延迟改进，而是提供一种使内存仲裁可观察、可归因的机制。
