## 摘要
大语言模型（LLM）彻底改变了AI应用，但大规模部署仍面临重大挑战。我们提出了RTP-LLM，一个用于工业级LLM部署的高性能推理引擎，已在阿里巴巴集团成功部署，服务超过1亿用户。RTP-LLM通过集成设计解决了关键瓶颈。它通过文件顺序驱动的I/O和并行I/O-通信重叠优化了模型加载。Prefill-Decode分离架构将计算密集的预填充阶段与内存受限的解码阶段解耦，结合分层多级KV缓存管理，实现了高效的缓存复用。此外，RTP-LLM集成了支持多种算法的模块化推测解码、自适应KV缓存量化以及解耦的多模态处理，并支持多层次并行。我们针对多种模型架构（8B-235B参数）进行了全面评估，使用了控制基准测试和真实生产工作负载。结果表明，与vLLM和SGLang相比，RTP-LLM表现出优越性能：模型加载速度提升4.7倍至6.3倍，生产流量调度中TTFT P95延迟降低35-37%，缓存复用改善215%，推测解码和多模态推理的吞吐量分别提升1.12倍至2.48倍和1.86倍至2.52倍，量化推理中批处理延迟降低35-40%，TTFT改善1.9倍至3.0倍。RTP-LLM经过生产验证的架构和开源可用性使其成为工业LLM部署的全面解决方案。

## Abstract
Large Language Models (LLMs) have revolutionized AI applications, but deploying them at scale presents significant challenges. We present RTP-LLM, a high-performance inference engine for industrial-scale LLM deployment, successfully deployed across Alibaba Group serving over 100 million users. RTP-LLM addresses fundamental bottlenecks through integrated design. It optimizes model loading via file-order-driven I/O and parallel I/O-communication overlapping. The Prefill-Decode Disaggregation architecture decouples compute-intensive prefill from memory-bound decode phases, combined with hierarchical multi-tiered KV cache management enabling efficient cache reuse. In addition, RTP-LLM incorporates modular speculative decoding supporting multiple algorithms, adaptive KV cache quantization, and decoupled multimodal processing, with support for multi-level parallelism.   Comprehensive evaluations across diverse model architectures (8B-235B parameters) have been conducted, where both controlled benchmarks and real production workloads are used. The results demonstrate RTP-LLM's superior performance against vLLM and SGLang: 4.7x-6.3x model loading speedup, 35-37% TTFT P95 latency reduction with 215% cache reuse improvement in production traffic scheduling, 1.12x-2.48x and 1.86x-2.52x throughput improvements in speculative decoding and multimodal inference, respectively, and 35-40% batch latency reduction with 1.9x-3.0x TTFT improvement in quantized inference. RTP-LLM's production-proven architecture and open-source availability make it a comprehensive solution for industrial LLM deployment.

## 1. 核心问题与整体含义（研究动机与背景）

- **核心问题**：大规模 LLM 生产部署面临四大挑战：  
  1. GPU 利用率低：动态顺序工作负载（可变输入/输出长度）导致计算单元闲置，静态批处理无法适应。  
  2. KV 缓存内存耗尽：随序列长度和批次大小线性增长，传统分配器碎片化严重。  
  3. 系统架构刚性：难以高效处理 MoE（>600B）、多模态等异构模型，缺乏灵活并行支持。  
  4. 操作脆弱性：模型加载耗时数小时（600B+ 参数），缺乏容错、滚动更新、请求隔离等生产级特性。  

- **整体含义**：现有系统（如 vLLM、SGLang）多聚焦孤立组件优化，忽视系统级交互，无法满足工业级动态、异构、高吞吐、低延迟的部署需求。RTP-LLM 旨在提供一个**集成化、生产验证**的推理引擎，已服务阿里巴巴超 1 亿用户。

## 2. 方法论：核心思想、关键技术细节

### 2.1 高效模型加载
- **核心思想**：从“模型结构驱动”转换为“文件顺序驱动”加载，避免冗余文件读取和非顺序访问。
- **关键技术**：
  - 文件顺序读写：按文件顺序加载所有张量，最大化 FUSE 预取效率。
  - 混合分布式读取：每个模型文件仅由一个进程读取，通过 PyTorch broadcast 共享张量，消除多进程重复读取。
  - 共享内存复用：重用单个 share memory buffer，避免每次文件读取重新分配（节省 600ms/2GB 开销）。
  - 并行 I/O-通信重叠：文件读取与张量广播并行执行，降低总加载时间。

### 2.2 Prefill-Decode 分离架构（PD Disaggregation）
- **核心思想**：物理上将计算密集的 Prefill 阶段（并行处理输入）与内存受限的 Decode 阶段（自回归生成）分离到独立节点，独立扩展资源。
- **关键技术**：
  - 动态负载均衡：Master 根据队列状态、KV 缓存占用、预测完成时间（公式 1）调度 Prefill 请求；Decode 请求优先路由到已有缓存 session 的 worker。
  - 智能批处理：窗口大小动态调整，分组相似长度请求以减少填充开销。

### 2.3 分层多级 KV 缓存管理
- **核心思想**：通过四层缓存层次（GPU→本地 CPU→远程 CPU RDMA→分布式存储 3FS），实现高效缓存复用。
- **关键技术**：
  - 统一哈希映射：合并所有 worker 的缓存键为单一大表，实现 O(B) 前缀匹配（避免 O(B×W)）。
  - 采样前缀哈希：对大块（≥208 tokens）在 208,212,216,… 位置创建哈希，平衡匹配粒度与存储开销。
  - 缓存亲和路由：Master 计算缓存复用得分（公式 2），结合负载决定调度，优先路由已有 session 的 worker。
  - 降级策略：GPU→本地→远程→3FS 逐级加载，配合 LRU 更新。

### 2.4 模块化推测解码
- **核心思想**：通过并行预测和验证多个未来 token，突破自回归顺序瓶颈。
- **关键技术**：
  - 四组件架构：ProposeExecutor（多种算法：Medusa、Eagle、Prompt Lookup、MTP）、ScoreExecutor（并行打分）、SpeculativeSampler（验证）、SpeculativeUpdater（更新状态）。
  - Prompt Lookup 算法（算法 3）：N-gram 匹配历史 prompt，提取 k 个候选，通过目标模型验证。特别适用于代码编辑场景（光标维护、跨步匹配优化）。
  - C++ 直接算子启动，消除 Python→C++ 调用开销。

### 2.5 自适应量化与多级并行
- **量化**：
  - 主要采用**权重独量量化**（AWQ/FP8），支持 FP8 KV Cache 量化（逐张量/逐块缩放），减少内存带宽压力。
- **并行策略**：
  - 张量并行（TP）、流水线并行（PP）、数据并行（DP）、专家并行（EP）组合，适应密集和 MoE 模型（如 Qwen3-Coder-480B-FP8 使用 TP=8, EP=8, DP=1）。

### 2.6 解耦多模态处理（EPD Disaggregation）
- **核心思想**：ViT 与 LLM 独立部署，分离视觉编码与语言生成，避免资源争用，实现计算重叠。
- **实现**：ViT 在独立 GPU 上处理图像生成 embedding，再与文本输入拼接送至 LLM 引擎。支持 LLaVA、Qwen-VL 等多模态架构。

### 2.7 系统架构（算法 1）
- 前端 → Master（生成哈希、前缀匹配、决策批处理） → 分层缓存加载（GPU→本地→远程→3FS） → 推理执行 → 结果返回并更新 LRU。

## 6. 主要结论与发现

- **模型加载**：相比 vLLM/SGLang 加速 4.7x-6.3x（Qwen3-235B 加载从 174s 降至 33s），且 TP 越高优势越大（SGLang/vLLM 负扩展）。

- **完整系统集成**：将加载、调度、缓存、量化、推测解码、多模态等优化统一到单一引擎，而非碎片化解决方案。
- **生产验证**：来自阿里巴巴的真实业务场景（超 1 亿用户），结果可靠，具有实际指导意义。
- **工程设计优秀**：C++ 直接算子启动避免 Python 开销；文件顺序加载、缓存亲和调度等巧妙设计。
- **开源可用**：提供了可扩展的基础，社区可进一步开发。
