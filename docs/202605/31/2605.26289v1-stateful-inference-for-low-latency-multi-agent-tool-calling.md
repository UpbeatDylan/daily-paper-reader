## 摘要
多智能体工具调用正成为基于LLM系统的主导交互模式，然而现有推理框架将每次工具调用视为独立请求，从头重新处理整个对话，尽管85-95%的提示与上一轮相比未发生变化。我们提出了一种有状态推理架构，将传统服务的每轮成本从O(n_t)降低为仅需处理增量的O(Δ_t)成本：持久化KV缓存跨轮次存活，仅通过摄入新令牌向前推进；同时，基数前缀缓存将其扩展到交织的多智能体流量中，而提示查找推测解码器加速了结构化输出。在全新生成的负载上，与vLLM和SGLang相比，参考实现在6轮智能体工作流中每轮速度快2.1倍，在35轮工作流的中位数轮次中快4.2倍，端到端wall time减半。这一优势源于有状态重用和推测，而非缓存。

## Abstract
Multi-agent tool calling is becoming the dominant interaction pattern for LLM-based systems, yet existing inference frameworks treat each tool call as an independent request, re-processing the entire conversation from scratch even though 85-95% of the prompt is unchanged from the previous turn.   We present a stateful inference architecture that converts the $O(n_t)$ per-turn cost of conventional serving into an $O(Δ_t)$ delta-only cost: a persistent KV cache lives across turns and advances by ingesting only the new tokens, while a radix prefix cache extends this across interleaved multi-agent traffic and a prompt-lookup speculative decoder accelerates structured output. Against vLLM and SGLang on novel, fully-generated workloads, the reference implementation is $2.1\times$ faster per turn on a 6-turn agentic workflow and $4.2\times$ on the median turn of a 35-turn one, halving end-to-end wall time. The advantage comes from stateful reuse and speculation, not caching.

# 论文详细总结：Stateful Inference for Low-Latency Multi-Agent Tool Calling

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：多智能体工具调用（multi-agent tool calling）正成为LLM系统的主流交互方式。在这类工作负载中，每次工具调用都会将新生成的令牌和工具结果附加到已有对话上，导致对话内容单调增长。然而现有推理框架（如vLLM、SGLang）将每次调用视作独立请求，从头重新处理整个对话（prefill），尽管85-95%的提示与上一轮完全相同。这造成了大量冗余计算，在典型5轮智能体对话中，最后一轮约950个令牌中仅有约150个是新令牌，而GPU集群常被报告利用率仅10-45%，大量浪费来自这种重复处理。
- **整体含义**：要降低延迟和计算成本，必须跨请求保留和重用中间状态，而非每次从零开始。论文提出了一个系统的有状态推理架构，从根本上解决这一开销问题。

## 2. 论文提出的方法论

### 核心思想
将跨请求的KV缓存持久化（stateful session），每个后续轮次仅处理新增的增量令牌（Δ_t），而非整个监督前缀。同时通过基数前缀缓存（radix prefix cache）实现跨多个智能体交织流量的高效共享，并通过提示查找推测解码（prompt-lookup speculative decoding）加速结构化工具调用输出。

### 关键技术细节
1. **有状态会话（Stateful Sessions）**：持久KV缓存跨请求存活，每轮仅摄入新增令牌。
2. **序列池（Sequence Pool）**：在统一推理运行上下文中预分配一组命名序列ID，通过线程安全的空闲列表回收，消除每请求上下文构造开销。
3. **基数前缀缓存（Radix Prefix Cache）**：以token序列为键的基数树，通过元数据性序列别名（metadata-only sequence aliasing）在统一KV缓存中共享前缀：将新序列的页表重写为指向捐赠者的页表，常数时间内恢复长为m的前缀，将每轮成本从O(n_t)转为O(Δ_t)。
4. **连续批处理调度器（Continuous Batching Scheduler）**：
   - **单元格预算准入（Cell-Budget Admission）**：跟踪统一KV缓存占用，仅当预计总token数低于特定上限时准入新prefill。
   - **前缀感知分组预填充（Prefix-Aware Grouped Prefill）**：将共享字节相同前缀的多个槽位合并为单个领导者-追随者预填充，将N个冗余前向传播压缩为1个。
5. **提示确定性响应缓存（Prompt-Deterministic Response Cache）**：通过FNV-1a哈希种子采样器，相同提示产生相同输出，对重复请求返回缓存结果（零GPU工作）。
6. **并发上限的提示查找推测解码（Concurrency-Capped Prompt-Lookup Speculation）**：利用槽位最近token历史预测后续token，再在单次前向传播中验证；通过自适应每槽限帽防止吞吐量崩溃。
7. **流式工具调用验证器（Streaming Tool-Call Validator）**：括号平衡JSON解析器在工具调用结构关闭时提前停止生成，过滤幻觉工具名，节省20-40%解码token。
8. **设备上贪心采样（On-Device Greedy Sampling）**：直接在加速器上执行argmax，消除每token同步-扫描往返延迟。
9. **RAII会话生命周期（RAII Session Lifecycle）**：异常安全防护自动获取/释放序列槽，防止并发多智能体工作负载下的泄漏。

### 成本模型公式（文字说明）
- **常规推理**：每轮成本 O(n_t)（全预填充）+ m次后向传播。
- **本文方法**：常数时间别名恢复 + O(Δ_t) 增量预填充 + ceil(m/(k+1))次后向传播（k为接受的推测token数，典型约11）。
- **重复提示**：缓存命中时HTTP往返延迟即可返回，无GPU工作。

## 3. 实验设计

## 6. 主要结论与发现

1. **架构优势**：将多智能体工具调用的每轮成本从O(n_t)降低到O(Δ_t)，通过状态复用和推测解码显著降低延迟。
2. **性能提升**：
   - 新颖工作流上比vLLM/SGLang快2.1-4.2倍。
   - 深度工作流端到端时间减半。
3. **优势随深度扩大**：对话越长，冗余越小，加速比越大。
4. **突发模式下p99尾延迟降低一个数量级**，得益于前缀共享避免独立预填充。
5. **解释性**：优势源于系统设计，而非缓存命中率；在零缓存命中场景下仍大幅度领先。
6. **对产业意义**：每个被消除的冗余prefill和每个被接受的推测窗口都是节省的FLOPs，可提升GPU集群利用率。

## 7. 优点

1. **系统性解决方案**：从序列池、前缀缓存、调度器、推测解码、验证器、采样等进行全栈协同设计。
2. **针对问题精准匹配**：深刻识别多智能体工作负载的结构特征（单调增长、高度结构化输出），并逐个机制对应。
3. **实验设计隔离贡献**：将响应缓存排除在主基准外，避免噪音，公平刻画推理加速。
4. **兼容性和实践性**：与现有API格式兼容，无需修改模型权重或训练过程。
5. **零额外GPU内存**：提示查找推测解码仅基于CPU token缓冲区，不消耗GPU显存。

## 8. 不足与局限

1. **实验覆盖有限**：仅测试8B模型和单L40S GPU；未评估更大模型（如70B）或多GPU分布式场景。
2. **消融实验不够**：未单独测量基数前缀缓存与推测解码的各自贡献；未与无状态但有前缀缓存的变体对比（例如与vLLM/SGLang前缀缓存直接比较）。
3. **硬件和能耗**：未报告能耗、功耗数据，也未涉及成本分析。
4. **应用范围限制**：前缀匹配是精确的，任何token差异都导致缓存未命中；不适用于提示或上下文修改场景（如对话历史被裁剪或重新排序）。
5. **内存开销**：缓存前缀消耗KV缓存容量，对于长对话和多并发智能体，需要足够大的上下文长度分配。在资源受限情况下可能面临压力。
